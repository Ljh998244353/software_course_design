# 在线拍卖平台物品与审核模块说明

## 1. 文档目的

本文档用于完成 `S05` 和后续 `S19` 前端真实接入，明确物品发布与审核模块在课程设计阶段的职责范围、业务流程、状态流转、接口契约、关键错误码、事务边界和依赖关系。

本模块是拍卖活动创建前的前置模块，负责把卖家提交的拍品从草稿状态推进到可创建拍卖活动的状态。

## 2. 模块职责

物品与审核模块负责以下能力：

- 卖家创建、保存和修改拍品草稿
- 卖家上传、删除和排序拍品图片
- 卖家提交拍品审核申请
- 管理员查看待审核拍品并执行通过或驳回
- 记录审核日志和驳回原因
- 维护拍品从草稿到待拍卖的生命周期状态

本模块不负责以下内容：

- 不负责创建拍卖活动
- 不负责竞价规则、最高价更新和订单生成
- 不负责支付、评价和统计业务规则
- 不决定拍卖活动状态，仅为拍卖模块提供“可上架拍卖”的拍品输入

## 3. 核心实体与依赖

### 3.1 核心实体

本模块核心实体如下：

- `Item`
- `ItemImage`
- `ItemAuditLog`
- `ItemCategory`

### 3.2 主要依赖

| 依赖对象 | 作用 | 边界说明 |
|---|---|---|
| 认证与权限模块 | 识别当前用户身份、角色和账号状态 | 所有写接口必须先完成登录校验和状态校验 |
| `item` 表 | 存储拍品主数据 | 是拍品状态的权威来源 |
| `item_image` 表 | 存储图片元数据 | 只保存图片地址和排序信息，不保存文件本体 |
| `item_audit_log` 表 | 存储审核轨迹 | 审核写入必须和拍品状态变更同事务 |
| 文件存储适配器 | 存储图片文件 | 上传成功后才能落图片元数据 |
| 通知模块 | 发送审核结果通知 | 通知失败不影响审核事务提交 |

## 4. 角色与权限边界

### 4.1 卖家权限

`USER` 在物品模块中具备以下能力：

- 创建拍品草稿
- 编辑自己名下的草稿或被驳回拍品
- 上传和删除自己拍品的图片
- 提交自己拍品进入审核
- 查询自己拍品列表、详情、审核状态和驳回原因

`USER` 不允许：

- 审核他人拍品
- 修改他人拍品
- 直接将拍品改为“审核通过”或“待拍卖”

### 4.2 管理员权限

`ADMIN` 具备以下能力：

- 查看待审核拍品列表
- 查看拍品详情和审核历史
- 审核通过拍品
- 驳回拍品并填写原因

`ADMIN` 不负责：

- 修改卖家原始标题、描述和图片内容
- 直接创建订单、触发支付、修改竞价记录

### 4.3 客服/运维权限

`SUPPORT` 默认不参与正式审核流转，可具备只读查询能力：

- 查看拍品详情
- 查看审核日志
- 查询图片与异常记录

`SUPPORT` 不允许：

- 审核通过或驳回拍品
- 修改拍品状态

## 5. 状态模型

### 5.1 拍品状态定义

`item.item_status` 建议采用以下状态：

- `DRAFT`：草稿，卖家可继续编辑，尚未提交审核
- `PENDING_AUDIT`：待审核，等待管理员处理
- `REJECTED`：审核驳回，卖家可查看原因并修改后重新提交
- `READY_FOR_AUCTION`：审核通过，可供拍卖管理模块创建活动
- `IN_AUCTION`：已被拍卖活动引用且处于竞拍期
- `SOLD`：拍品已成交
- `UNSOLD`：拍品流拍
- `OFFLINE`：人工下架或不再展示

### 5.2 状态流转规则

```text
DRAFT -> PENDING_AUDIT
PENDING_AUDIT -> READY_FOR_AUCTION
PENDING_AUDIT -> REJECTED
REJECTED -> DRAFT
READY_FOR_AUCTION -> IN_AUCTION
IN_AUCTION -> SOLD
IN_AUCTION -> UNSOLD
READY_FOR_AUCTION -> OFFLINE
UNSOLD -> OFFLINE
```

状态说明如下：

- 卖家提交审核时，`DRAFT` 才能转为 `PENDING_AUDIT`
- 管理员审核通过后，`PENDING_AUDIT` 转为 `READY_FOR_AUCTION`
- 管理员驳回后，`PENDING_AUDIT` 转为 `REJECTED`
- 卖家修改驳回拍品时，先回退为 `DRAFT`，再允许再次提交审核
- 一旦进入 `IN_AUCTION`，物品与审核模块不再允许卖家直接修改核心字段

### 5.3 审核结果定义

`item_audit_log.audit_result` 建议如下：

- `APPROVED`
- `REJECTED`

## 6. 业务流程

### 6.1 卖家创建草稿流程

1. 卖家登录后进入拍品发布页。
2. 系统校验账号状态必须为 `ACTIVE`。
3. 卖家填写标题、描述、分类、建议起拍价等字段。
4. 系统保存 `item` 记录，状态设为 `DRAFT`。
5. 卖家可后续补充图片并继续编辑。

### 6.2 图片上传流程

1. 卖家对自己的草稿或驳回拍品上传图片。
2. 系统校验文件类型、大小、数量和扩展名。
3. 文件上传到本地文件存储或对象存储。
4. 上传成功后，系统写入 `item_image` 元数据。
5. 如上传失败，则不写入数据库元数据。

### 6.3 提交审核流程

1. 卖家对 `DRAFT` 状态拍品发起提交审核。
2. 系统校验标题、描述、分类、图片、建议起拍价等字段完整性。
3. 系统校验当前用户是拍品所有者且账号状态正常。
4. 系统更新 `item_status = PENDING_AUDIT`。
5. 系统可记录业务日志，供管理员待审列表查询。

### 6.4 管理员审核流程

1. 管理员查看待审核列表。
2. 管理员进入拍品详情，检查标题、描述、图片和分类信息。
3. 管理员选择通过或驳回，并填写审核意见。
4. 系统在同一事务中更新拍品状态并插入 `item_audit_log`。
5. 事务提交成功后，异步发送审核结果通知。

### 6.5 驳回后重提流程

1. 卖家查看驳回原因。
2. 卖家修改标题、描述、图片或分类。
3. 系统将状态调整为 `DRAFT`。
4. 卖家重新提交审核，进入下一轮审核流转。

## 7. 接口设计

所有接口遵循统一响应结构：

```json
{
  "code": 0,
  "message": "success",
  "data": {}
}
```

### 7.1 `POST /api/items`

用途：

- 卖家创建拍品草稿

鉴权要求：

- 需要登录
- 仅 `USER`

请求字段：

| 字段 | 类型 | 必填 | 说明 |
|---|---|---|---|
| title | string | 是 | 拍品标题 |
| description | string | 是 | 拍品描述 |
| categoryId | number | 是 | 分类 ID |
| startPrice | number | 是 | 建议起拍价 |
| coverImageUrl | string | 否 | 封面图地址 |

响应字段：

| 字段 | 类型 | 说明 |
|---|---|---|
| itemId | number | 新建拍品 ID |
| itemStatus | string | 默认 `DRAFT` |
| createdAt | string | 创建时间 |

### 7.2 `PUT /api/items/{id}`

用途：

- 卖家编辑自己的草稿或驳回拍品

鉴权要求：

- 需要登录
- 仅拍品所有者

允许编辑状态：

- `DRAFT`
- `REJECTED`

请求字段：

| 字段 | 类型 | 必填 | 说明 |
|---|---|---|---|
| title | string | 否 | 标题 |
| description | string | 否 | 描述 |
| categoryId | number | 否 | 分类 |
| startPrice | number | 否 | 建议起拍价 |
| coverImageUrl | string | 否 | 封面图 |

响应字段：

| 字段 | 类型 | 说明 |
|---|---|---|
| itemId | number | 拍品 ID |
| itemStatus | string | 更新后的状态 |
| updatedAt | string | 更新时间 |

错误场景：

- 非拍品所有者
- 当前状态不允许编辑
- 字段校验不通过

### 7.3 `POST /api/items/{id}/images`

用途：

- 为拍品上传图片

鉴权要求：

- 需要登录
- 仅拍品所有者

允许状态：

- `DRAFT`
- `REJECTED`

请求字段：

| 字段 | 类型 | 必填 | 说明 |
|---|---|---|---|
| file | binary | 是 | 图片文件 |
| sortNo | number | 否 | 排序号 |
| isCover | boolean | 否 | 是否封面 |

响应字段：

| 字段 | 类型 | 说明 |
|---|---|---|
| imageId | number | 图片 ID |
| imageUrl | string | 图片地址 |
| sortNo | number | 排序号 |
| isCover | boolean | 是否封面 |

错误场景：

- 文件类型非法
- 文件大小超限
- 图片数量超限
- 当前状态不允许上传图片

### 7.4 `DELETE /api/items/{itemId}/images/{imageId}`

用途：

- 删除拍品图片

鉴权要求：

- 需要登录
- 仅拍品所有者

说明：

- 删除图片时应先删除数据库元数据，再对存储文件做最佳努力清理
- 若物理文件删除失败，可进入异步清理任务，但不应阻塞主流程

### 7.5 `POST /api/items/{id}/submit-audit`

用途：

- 卖家提交审核

鉴权要求：

- 需要登录
- 仅拍品所有者

允许状态：

- `DRAFT`

请求字段：

- 无额外业务体

响应字段：

| 字段 | 类型 | 说明 |
|---|---|---|
| itemId | number | 拍品 ID |
| itemStatus | string | 更新后为 `PENDING_AUDIT` |
| submittedAt | string | 提交时间 |

错误场景：

- 字段不完整
- 图片缺失
- 当前状态不是 `DRAFT`
- 当前用户不是拍品所有者

## 8. 事务边界与一致性要求

### 8.1 草稿创建与编辑

事务边界：

- `item` 主记录写入在单事务中完成
- 图片元数据更新与拍品主记录更新应在同一事务中提交

说明：

- 图片文件本体上传属于外部存储操作，不与数据库共享事务
- 若图片文件已上传但数据库提交失败，应做最佳努力回滚或记录待清理文件

### 8.2 提交审核

事务边界：

- `item_status` 从 `DRAFT` 更新为 `PENDING_AUDIT` 在单事务内完成

校验要求：

- 必须检查拍品所有权
- 必须检查字段完整性和图片完整性
- 必须检查账号状态为 `ACTIVE`

### 8.3 管理员审核

事务边界：

- 更新 `item.item_status`
- 写入 `item.reject_reason`
- 插入 `item_audit_log`

上述三项必须在同一事务中成功或失败。

通知要求：

- 审核结果通知在事务提交成功后发送
- 通知失败不回滚审核结果，只记录失败并允许补发

## 9. 错误码设计

物品与审核模块建议使用 `4200-4299` 段：

| 错误码 | 说明 |
|---|---|
| 4201 | 拍品不存在 |
| 4202 | 当前用户不是拍品所有者 |
| 4203 | 当前状态不允许编辑 |
| 4204 | 当前状态不允许提交审核 |
| 4205 | 当前状态不允许执行审核 |
| 4206 | 分类不存在或不可用 |
| 4207 | 图片上传失败 |
| 4208 | 图片数量或格式不合法 |
| 4209 | 驳回原因不能为空 |
| 4210 | 审核结果非法 |

同时可复用全局错误码：

- `4001`：参数非法
- `4002`：资源不存在
- `4004`：权限不足
- `5001`：内部服务异常
