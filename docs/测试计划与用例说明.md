# 在线拍卖平台测试计划与用例说明

## 1. 文档目的

本文档用于记录 `S13` 自动化测试基线的真实落地结果，统一说明当前仓库的测试分层、执行入口、覆盖边界和后续专项测试衔接方式。

`S13` 的目标是基于现有工程做统一收口，不重复引入与当前代码实现不匹配的空测试骨架。

## 2. 当前测试基线

当前仓库的自动化测试基线如下：

- 使用 `CTest` 作为统一注册、筛选和执行入口。
- 测试程序主体采用“独立测试二进制 + `assert` 断言”的方式，而不是额外平移到新的测试框架。
- 所有依赖数据库的测试继续复用 `scripts/db/setup_local_mysql.sh` 和 `scripts/db/write_test_config.sh`。
- `scripts/test_*.sh` 负责单模块或专项测试的数据库准备与测试启动。
- `scripts/test.sh` 作为统一总入口，按测试分层或模块标签调用 `ctest`。

需要说明：

- 课程计划中曾写到 `GoogleTest`，但当前代码现实并未采用该框架。
- 本步以“统一现有测试基线”为目标，因此保留已稳定运行的 `CTest + 断言式测试二进制 + Shell 脚本` 方案。

## 3. 目录与分层

当前测试目录按以下方式组织：

| 分层 | 代码位置 | 说明 |
|---|---|---|
| 单元/冒烟测试 | `tests/unit/smoke_tests.cpp` | 校验配置模板、统一响应、错误码和健康检查基础能力 |
| 数据库冒烟测试 | `tests/integration/database_smoke_tests.cpp` | 校验 schema、seed、应用连库和 `--check-db` |
| 模块流测试 | `tests/auth/` `tests/item/` `tests/auction/` `tests/bid/` `tests/order/` `tests/payment/` `tests/review/` `tests/statistics/` `tests/ops/` | 每个模块各自验证最小业务闭环 |
| 主流程联调/契约测试 | `tests/integration/full_flow_tests.cpp` | 覆盖认证、拍品、审核、拍卖、竞价、订单、支付、履约、评价、统计、运维的服务层联调闭环 |
| 测试脚本 | `scripts/test.sh` `scripts/test_*.sh` | 统一入口与分模块入口 |

当前未单独建立 `tests/security/` 或 `tests/stress/`，对应高风险专项测试放到后续 `S14` 落地，避免在 `S13` 阶段先创建空目录。

## 4. CTest 标签与执行入口

`S13` 新增并统一了以下 `CTest` 标签：

| 标签 | 作用 |
|---|---|
| `suite_smoke` | 运行不依赖 MySQL 的基础冒烟测试 |
| `suite_db` | 运行数据库冒烟测试 |
| `suite_module` | 运行各业务模块最小闭环测试 |
| `suite_integration` | 运行跨模块集成验证 |
| `suite_contract` | 运行当前接口契约映射验证，现阶段对应主流程联调 |
| `requires_mysql` | 标记依赖本地测试 MySQL 的用例 |
| `module_auth` 等模块标签 | 支持按业务模块单独执行 |

统一执行入口如下：

```bash
scripts/test.sh
scripts/test.sh smoke
scripts/test.sh db
scripts/test.sh module
scripts/test.sh integration
scripts/test.sh contract
scripts/test.sh auth
scripts/test.sh list
```

`scripts/test.sh` 内部会先执行 `scripts/bootstrap.sh`，再通过 `ctest` 和标签筛选执行对应测试集。

如需直接使用 `ctest`，当前支持的常用方式如下：

```bash
ctest --test-dir build --output-on-failure
ctest --test-dir build --output-on-failure -L suite_module
ctest --test-dir build --output-on-failure -L suite_contract
ctest --test-dir build --output-on-failure -L module_bid
```

## 5. 共享 MySQL 运行时约束

当前大部分业务测试都依赖 `build/test_mysql` 下的本地 MySQL 运行时目录。

为避免 `ctest -j` 或按标签回归时多个测试同时竞争同一套 `datadir`、`socket` 和 `ibdata1` 文件，`S13` 在 `CMakeLists.txt` 中为所有依赖数据库的用例统一增加了：

- `requires_mysql` 标签
- `RESOURCE_LOCK auction_mysql_test_runtime`

这样即使使用并行 `ctest`，依赖同一本地 MySQL 的测试也会按资源锁顺序执行，减少互相踩库导致的伪失败。

## 6. 覆盖范围

当前自动化测试已经覆盖以下内容：

- 工程配置加载、统一响应模型、错误码与健康检查。
- 数据库 schema、seed、基础连库验证。
- 认证、物品审核、拍卖管理、竞价通知、订单支付、评价、统计、运维各模块最小闭环。
- 支付后履约、互评、统计重建和运维查询的主流程联调。
- 当前服务层实现所对应的接口契约映射验证。

当前明确未覆盖或只做最小覆盖的内容：

- Drogon HTTP 控制器批量接入后的真实接口级自动化测试。
- 并发出价、拍卖结束竞争、支付回调幂等回归、Redis 降级、通知失败、安全负向等高风险专项测试。
- 200 并发在线验证与性能测试。

以上内容统一留到 `S14` 继续落地。

`S13` 实际执行并通过以下验证命令：

```bash

scripts/test.sh db
scripts/test.sh module

```

验证结论如下：

- MySQL 依赖测试已统一挂载资源锁。
- 当前全量 `ctest` 为 `13/13` 全部通过。
