# 在线拍卖平台用户使用手册

## 1. 文档说明

本文档面向系统最终使用者，覆盖以下四类角色的完整操作说明：

- 普通用户：同一账号可同时作为买家和卖家使用
- 管理员：负责拍品审核、拍卖管理、统计查看和用户治理
- 运维/客服：负责日志查询、异常排查、通知重试和补偿协助
- 演示人员：按本文档可完整走通真实产品演示流程

本文档只描述“怎么使用系统”，不展开底层实现细节。

## 2. 系统访问

### 2.1 访问地址

- 前端页面：`http://127.0.0.1:3000`
- 后端 API：`http://127.0.0.1:18080`
- 健康检查：`http://127.0.0.1:18080/healthz`

### 2.2 主要页面

| 页面 | 路由 | 适用角色 | 作用 |
|---|---|---|---|
| 首页 | `/` | 全部角色 | 浏览公开拍卖、搜索、进入大厅 |
| 登录页 | `/auth/login` | 全部角色 | 登录、注册、恢复会话 |
| 拍卖大厅 | `/auction/hall` | 普通用户 | 按条件筛选拍卖并进入详情 |
| 拍卖详情 | `/auction/detail/[id]` | 普通用户 | 查看实时价格、出价历史、参与竞价 |
| 发布拍品 | `/auction/publish` | 普通用户 | 创建拍品草稿、上传图片、提交审核 |
| 支付页 | `/checkout/[orderId]` | 普通用户 | 查看订单、发起测试支付 |
| 管理大盘 | `/admin/dashboard` | 管理员 | 审核拍品、查看统计、查看待处理事项 |

说明：

- 普通用户没有单独的“卖家账号”与“买家账号”之分，同一账号可以发布拍品，也可以参与竞价。
- 支付链路使用测试支付模型，但页面操作、状态推进、通知和审计逻辑与正式产品一致。

## 3. 角色与权限

| 角色 | 可执行操作 | 不可执行操作 |
|---|---|---|
| 普通用户 | 注册、登录、发布拍品、查看审核结果、浏览拍卖、出价、支付、确认收货、互评 | 审核拍品、管理用户、查看后台运维日志 |
| 管理员 | 审核拍品、查看待审列表、查看日报统计、管理用户状态、处理后台管理任务 | 不能绕过正常业务直接替用户出价或代付 |
| 运维/客服 | 查看操作日志、任务日志、异常列表、重试通知、执行补偿、协助排障 | 审核拍品、批准/驳回拍品、修改用户角色 |

## 5. 开始使用前的共识

### 5.1 业务状态简表

拍品状态：

- `DRAFT`：草稿
- `PENDING_REVIEW`：待审核
- `APPROVED`：审核通过
- `REJECTED`：审核驳回

拍卖状态：

- `DRAFT` / `SCHEDULED`：待开始
- `RUNNING`：竞拍中
- `SETTLING`：待结算
- `SOLD`：已成交
- `UNSOLD`：流拍
- `CANCELED`：已取消

订单状态：

- `PENDING_PAYMENT`：待支付
- `PAID`：已支付待发货
- `SHIPPED`：已发货待收货
- `COMPLETED`：已完成待互评或已完成
- `CLOSED`：已关闭

### 5.2 实时刷新机制

- 详情页优先使用 WebSocket：`/ws/auction/{id}`
- 如果实时连接不可用，系统会自动降级为轮询刷新
- 降级时页面会提示“实时通道已断开”或类似横幅，但不影响继续查看和出价

## 6. 普通用户教程

### 6.1 注册与登录

#### 6.1.1 注册新账号

1. 打开首页或直接进入 `/auth/login`
2. 在登录页切换到注册表单
3. 填写用户名、密码、手机号
4. 提交注册
5. 注册成功后系统自动建立会话并进入拍卖大厅

建议：

- 用户名保持唯一，便于后续在订单、评价和后台审计中识别
- 密码需满足系统复杂度要求，避免使用过于简单的字符串

#### 6.1.2 登录

1. 进入 `/auth/login`
2. 输入用户名和密码
3. 点击登录
4. 登录成功后进入系统主页面

#### 6.1.3 登出

1. 在导航栏打开用户菜单
2. 点击登出
3. 系统清除当前登录态并回到公开页面

### 6.2 作为卖家使用

#### 6.2.1 发布拍品

入口：`/auction/publish`

操作步骤：

1. 填写拍品标题
2. 填写拍品描述
3. 选择分类
4. 填写封面图 URL 或上传图片元数据
5. 填写起拍价
6. 填写加价幅度
7. 设定开始时间与结束时间
8. 检查信息后提交

注意：

- 提交审核前至少要有一张图片
- 起拍价、加价幅度、时间范围必须满足页面和后端校验
- 首次提交时系统会先创建草稿，再写入图片，再提交审核

#### 6.2.2 查看拍品审核结果

当前前端重点入口仍是发布页和管理页联动，卖家需要重点关注以下结果：

- 审核通过：拍品可被管理员纳入拍卖活动
- 审核驳回：需根据驳回原因修改后重新提交

建议操作：

1. 重新进入发布页，按驳回原因修正标题、描述、图片或价格
2. 再次提交审核

#### 6.2.3 关注成交结果

卖家发布的拍品进入拍卖后，需要关注：

- 是否有人出价
- 拍卖结束后是否成交
- 买家是否完成支付

如果拍卖成交且买家已支付，后续进入履约流程。

#### 6.2.4 确认发货

卖家在订单进入 `PAID` 状态后执行发货。

标准流程：

1. 打开对应订单详情
2. 核对订单号、拍品标题、成交金额、买家信息
3. 点击确认发货
4. 订单进入 `SHIPPED`
5. 系统向买家发送“待确认收货”通知

#### 6.2.5 评价买家

订单完成后，卖家可提交对买家的评价。

建议内容：

- 履约是否及时
- 沟通是否顺畅
- 是否存在恶意拖延或异常行为

### 6.3 作为买家使用

#### 6.3.1 浏览拍卖

入口：

- 首页 `/`
- 拍卖大厅 `/auction/hall`

可执行操作：

- 搜索关键词
- 按拍卖状态筛选
- 按卖家信用筛选
- 按是否有历史成交筛选

#### 6.3.2 查看拍卖详情

入口：`/auction/detail/[id]`

页面重点信息：

- 当前最高价
- 起拍价和加价幅度
- 剩余时间
- 卖家信息和信用摘要
- 出价历史
- 实时通道状态

#### 6.3.3 参与竞价

1. 确认拍卖处于进行中
2. 查看当前价和最小加价幅度
3. 输入或选择新的出价金额
4. 点击 `PLACE BID`
5. 等待系统返回结果

可能结果：

- 成功：你成为当前最高出价者
- `409 Conflict`：他人先一步出价，当前价格已变化
- `429 Too Many Requests`：出价频率过高，被限流冷却
- 拍卖已结束：不能再出价
- 账号被冻结/禁用：不能继续执行写操作

#### 6.3.4 观察实时更新

详情页会实时反映：

- 最新价格
- 最新最高出价者状态
- 拍卖是否被延时保护顺延
- 你是否被其他买家超越

若 WebSocket 失败：

- 页面自动降级轮询
- 仍可继续刷新和出价
- 只是更新速度可能略慢

#### 6.3.5 支付订单

当你成为获胜买家后，系统会生成订单。

入口：`/checkout/[orderId]`

操作步骤：

1. 打开订单支付页
2. 核对订单号、拍品标题、成交金额
3. 选择测试支付方式：
   - `MOCK_WECHAT`
   - `MOCK_ALIPAY`
4. 点击确认支付
5. 等待支付状态更新

说明：

- 当前为测试支付模型，不会连真实第三方网关
- 但支付发起、回调、金额校验、幂等处理、通知和审计日志都是真实业务链路

#### 6.3.6 确认收货

在卖家发货后：

1. 打开订单详情
2. 核对订单状态为 `SHIPPED`
3. 点击确认收货
4. 订单进入 `COMPLETED`

#### 6.3.7 评价卖家

订单完成后，买家可以评价卖家：

1. 进入已完成订单
2. 选择评分
3. 填写评价内容
4. 提交评价

建议评价关注：

- 发货是否及时
- 拍品描述是否真实
- 交易沟通是否顺畅

## 7. 管理员教程

### 7.1 登录后台

使用 `admin` 账号登录后，进入 `/admin/dashboard`。

当前管理员前端已接入的核心功能：

- 待审核拍品列表
- 批准拍品
- 驳回拍品
- 日报统计查询

### 7.2 审核拍品

入口：管理大盘待审核区

审核步骤：

1. 查看待审拍品列表
2. 打开目标拍品
3. 核对标题、图片、分类、价格和描述
4. 执行批准或驳回

批准后：

- 拍品进入审核通过状态
- 可进入后续拍卖配置流程

驳回后：

- 必须填写原因
- 卖家按原因修改后可重新提交

### 7.3 查看日报统计

入口：`/admin/dashboard`

当前可查看数据：

- 指定日期范围内拍卖数量
- 成交数量
- 流拍数量
- 出价次数
- 成交金额

适用场景：

- 每日经营汇总
- 课堂演示 KPI
- 审核活动活跃度

### 7.4 用户治理

管理员还具备后端能力：

- 查看用户列表
- 修改用户状态
- 冻结违规账号
- 禁用严重违规账号

治理原则：

- `FROZEN`：通常用于临时限制
- `DISABLED`：通常用于永久禁用

冻结或禁用后：

- 旧会话会失效
- 用户后续登录或写操作会被拒绝

### 7.5 管理边界

管理员不应直接做以下事情：

- 替用户出价
- 替买家支付
- 强制伪造支付成功
- 直接绕过状态机写终态

## 8. 运维/客服教程

### 8.1 登录后台

使用 `support` 账号登录系统后，可进入管理相关页面和运维接口范围。

运维/客服重点不是拍品审核，而是排障和协助处理。

### 8.2 可执行操作

运维/客服可使用的典型能力包括：

- 查看操作日志
- 查看任务日志
- 查看异常记录
- 重试通知发送
- 发起补偿任务
- 协助定位订单、支付、竞价纠纷

### 8.3 操作日志查询

重点查看：

- 用户登录/登出
- 管理员审核动作
- 支付发起和回调
- 订单发货和确认收货

适合用于：

- 争议排查
- 课堂演示审计能力
- 回放关键操作链路

### 8.4 任务日志查询

重点查看：

- 拍卖开始/结束调度
- 统计汇总任务
- 支付关闭任务
- 补偿重试任务

适合用于：

- 查定时任务是否执行
- 查失败重试原因
- 查补偿是否完成

### 8.5 异常处理与通知重试

运维/客服可处理的常见场景：

- 某条通知未成功送达
- 某个业务主流程成功，但通知侧失败
- 某个订单或支付记录需要核对异常状态

处理思路：

1. 先查操作日志和任务日志
2. 再看异常记录
3. 需要时执行通知重试
4. 仍无法闭环时提交管理员或研发进一步处理

### 8.6 权限边界

运维/客服不能执行：

- 审核拍品通过/驳回
- 变更用户角色
- 直接修改拍卖业务事实

## 9. 全功能主流程演示

如果需要完整演示整个平台，建议按以下顺序：

1. 使用普通用户注册或登录 `seller_demo`
2. 发布一个新拍品并提交审核
3. 使用 `admin` 登录并审核通过
4. 等待该拍品进入可竞拍状态
5. 使用 `buyer_demo` 登录进入详情页出价
6. 观察 WebSocket 实时价格更新或轮询降级提示
7. 拍卖结束后生成订单
8. 使用 `buyer_demo` 进入支付页完成测试支付
9. 使用卖家确认发货
10. 使用买家确认收货
11. 买卖双方分别完成评价
12. 使用 `admin` 查看统计汇总
13. 使用 `support` 查看日志和异常处理入口

## 10. 常见问题

### 10.1 账号与登录

**Q：登录提示用户名或密码错误？**  
A：检查用户名拼写、大小写和密码是否正确。

**Q：登录提示账号已冻结？**  
A：管理员已将账号置为 `FROZEN`，需要联系管理员解冻。

**Q：登录提示账号已禁用？**  
A：账号已被置为 `DISABLED`，当前不可继续使用。

### 10.2 发布与审核

**Q：提交审核失败，提示至少需要一张图片？**  
A：请先补齐图片元数据，再重新提交。

**Q：拍品被驳回后怎么办？**  
A：根据驳回原因修改后重新提交审核。

### 10.3 竞价

**Q：为什么我出价后提示 409？**  
A：说明其他买家已经先提交了更早的有效出价，页面价格需要以最新状态为准。

**Q：为什么提示 429？**  
A：说明出价过于频繁，被系统限流，请等待冷却后再试。

**Q：页面提示实时通道断开怎么办？**  
A：系统会自动降级为轮询刷新，不影响继续使用。

### 10.4 支付与订单

**Q：为什么支付页显示的是测试支付？**  
A：当前系统保留测试支付模型用于全量演示，但订单、回调、状态机和审计逻辑都是真实链路。

**Q：超时未支付会怎样？**  
A：订单会关闭，拍卖会按业务规则进入关闭或流拍分支。

**Q：支付成功后状态没立刻刷新？**  
A：可能是回调或页面刷新存在短暂延迟，稍后重试即可。

### 10.5 评价

**Q：为什么我不能评价？**  
A：只有订单完成后，且你是买家或卖家本人，才能提交评价。

## 11. 使用建议

- 普通用户先熟悉“浏览拍卖 -> 出价 -> 支付 -> 收货 -> 评价”链路
- 卖家演示时重点突出“发布 -> 审核 -> 成交 -> 发货”
- 管理员演示时重点突出“审核 -> 统计 -> 用户治理”
- 运维/客服演示时重点突出“日志 -> 异常 -> 通知重试/补偿”
