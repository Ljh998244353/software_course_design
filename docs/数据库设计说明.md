# 在线拍卖平台数据库设计说明

## 1. 文档目的

本文档用于完成 `S03`，在现有需求规格和概要设计的基础上，给出本项目在课程设计阶段的数据库物理设计建议。

本步重点解决以下问题：

- 核心表如何落到 MySQL 8
- 关键状态字段如何定义
- 唯一约束和索引如何支撑主业务流程
- 初始化脚本如何组织
- 初始化数据和迁移策略如何规划

本文档是后续 `S04-S11` 模块设计与实现的数据库基线。

## 2. 设计约束

数据库设计遵循以下约束：

- 事务型事实数据统一落 MySQL，Redis 仅作缓存和热点读模型
- 课程设计阶段优先保证正确性、可验证性和可演示性，不引入分库分表
- 关键业务主链路必须能支撑用户注册登录、拍品发布审核、活动创建、出价竞拍、成交支付、评价反馈
- 拍卖、出价、订单、支付等关键状态必须可追踪
- 幂等、审计、补偿所需字段和日志表必须预留

## 3. 物理设计基线

### 3.1 数据库与字符集

建议基线如下：

- 数据库：MySQL 8
- 存储引擎：InnoDB
- 字符集：`utf8mb4`
- 排序规则：`utf8mb4_0900_ai_ci`
- 时间字段：优先使用 `datetime(3)`
- 金额字段：统一使用 `decimal(12,2)`
- 主键：统一使用 `bigint unsigned`

### 3.2 命名约定

建议约定如下：

- 表名使用小写下划线风格
- 主键统一为 `{table}_id` 或明确业务主键名
- 业务编号字段使用 `*_no`
- 创建时间统一为 `created_at`
- 更新时间统一为 `updated_at`
- 逻辑状态字段统一使用 `status`

说明：

- 概要设计中的 `User` 实体在物理表中建议使用 `user_account`，避免与系统库对象混淆
- 概要设计中的 `Order` 实体保持使用 `order_info`

### 5.6 `auction`

用途：

- 存储拍卖活动主状态
- 是竞价一致性的核心聚合根

| 字段名 | 类型 | 约束 | 说明 |
|---|---|---|---|
| auction_id | bigint unsigned | PK, auto_increment | 活动主键 |
| item_id | bigint unsigned | not null, FK -> item.item_id | 关联拍品 |
| seller_id | bigint unsigned | not null, FK -> user_account.user_id | 卖家 |
| start_time | datetime(3) | not null | 开始时间 |
| end_time | datetime(3) | not null | 结束时间 |
| start_price | decimal(12,2) | not null | 起拍价 |
| current_price | decimal(12,2) | not null | 当前最高价 |
| bid_step | decimal(12,2) | not null | 最小加价幅度 |
| highest_bidder_id | bigint unsigned | FK -> user_account.user_id | 当前最高出价者 |
| status | varchar(20) | not null | 活动状态 |
| anti_sniping_window_seconds | int | not null, default 0 | 触发延时保护的窗口秒数 |
| extend_seconds | int | not null, default 0 | 每次延长秒数 |
| version | bigint unsigned | not null, default 0 | 乐观锁版本号 |
| created_at | datetime(3) | not null | 创建时间 |
| updated_at | datetime(3) | not null | 更新时间 |

状态建议：

- `status`：`PENDING_START`、`RUNNING`、`SETTLING`、`SOLD`、`UNSOLD`、`CANCELLED`

说明：

- 为支持支付前结算控制，建议使用 `SETTLING` 作为拍卖结束到订单完成创建之间的过渡状态
- 同一拍品是否允许再次发起拍卖由业务规则决定，因此不对 `item_id` 做全局唯一约束
- 如需限制同一拍品同时仅存在一个非终态活动，建议在应用层事务里校验；MySQL 8 不适合用局部唯一索引直接表达该规则

索引建议：

- `idx_auction_status_start_end(status, start_time, end_time)`
- `idx_auction_item_status(item_id, status)`
- `idx_auction_seller_status(seller_id, status)`
- `idx_auction_highest_bidder(highest_bidder_id)`

### 5.7 `bid_record`

用途：

- 记录所有有效出价
- 作为竞价历史和一致性审计凭据

| 字段名 | 类型 | 约束 | 说明 |
|---|---|---|---|
| bid_id | bigint unsigned | PK, auto_increment | 出价主键 |
| auction_id | bigint unsigned | not null, FK -> auction.auction_id | 活动 |
| bidder_id | bigint unsigned | not null, FK -> user_account.user_id | 出价人 |
| request_id | varchar(64) | not null | 客户端幂等请求号 |
| bid_amount | decimal(12,2) | not null | 出价金额 |
| bid_status | varchar(20) | not null | 出价状态 |
| bid_time | datetime(3) | not null | 出价时间 |
| created_at | datetime(3) | not null | 记录创建时间 |

状态建议：

- `bid_status`：`WINNING`、`OUTBID`

关键约束：

- `uk_bid_record_request(auction_id, bidder_id, request_id)`

索引建议：

- `idx_bid_record_auction_created(auction_id, created_at)`
- `idx_bid_record_auction_bidder_created(auction_id, bidder_id, created_at)`

说明：

- 为保持历史清晰，建议只持久化有效出价，非法出价直接返回错误，不写入 `bid_record`
- 新最高价产生时，上一条最高出价记录应在同一事务中从 `WINNING` 更新为 `OUTBID`

## 7. 关键约束汇总

必须明确以下业务约束：

- `order_info.auction_id` 全局唯一，确保一场拍卖最多只生成一笔订单
- `payment_record.transaction_no` 全局唯一，确保第三方流水不重复入账
- `bid_record(auction_id, bidder_id, request_id)` 唯一，控制客户端重复提交
- `review(order_id, reviewer_id)` 唯一，防止重复评价
- `item_image(item_id, sort_no)` 唯一，防止同一拍品图片顺序冲突

必须通过事务或应用层控制的约束：

- 同一拍卖任一时刻只能存在一个合法当前最高价
- 出价记录写入、最高价更新、最高出价者更新、延时保护更新必须同事务提交
- 拍卖结束后不得再接受有效出价
- 同一拍品同时只能存在一个非终态拍卖活动

## 8. 索引策略汇总

高优先级索引如下：

- `auction(status, start_time, end_time)`
- `bid_record(auction_id, created_at)`
- `bid_record(auction_id, bidder_id, created_at)`
- `order_info(buyer_id, order_status)`
- `payment_record(order_id, pay_status)`
- `payment_record(transaction_no)`
- `notification(user_id, read_status, created_at)`
- `item(seller_id, item_status, created_at)`

说明：

- 索引优先服务活动调度、出价历史、订单查询、支付回调幂等和通知中心
- 不建议在课程设计阶段过早增加过多组合索引，避免写放大和维护复杂度

## 9. 初始化脚本规划

建议后续在 `sql/` 目录下按版本组织脚本：

```text
sql/
├── V001__create_core_tables.sql
├── V002__create_indexes_and_constraints.sql
├── V003__seed_base_data.sql
└── V004__seed_demo_data.sql
```

各脚本职责如下：

- `V001__create_core_tables.sql`：创建核心表和基础外键
- `V002__create_indexes_and_constraints.sql`：补充唯一约束、查询索引和必要检查
- `V003__seed_base_data.sql`：插入管理员账号、客服账号、分类字典、系统基础配置
- `V004__seed_demo_data.sql`：插入课程演示用测试用户、拍品和活动数据

脚本约定如下：

- 一次结构变更只对应一个新版本脚本，不直接覆盖旧脚本
- 演示数据与基础数据分离，防止污染正式初始化
- 所有脚本执行顺序固定，便于环境复现和评分答辩

## 10. 初始化数据建议

建议基础初始化数据至少包括：

- 一个管理员账号
- 一个客服或运维账号
- 若干普通测试账号
- 若干拍品分类
- 必要的系统配置项

说明：

- 不建议在基础种子数据里插入大量交易数据
- 出价记录、订单记录、支付记录更适合作为演示数据单独导入

## 11. 迁移策略

课程设计阶段建议采用前向兼容、版本递增的简单迁移策略：

1. 新增字段优先使用可空或带默认值方式上线
2. 每次结构变更必须新增一个版本脚本
3. 已提交的旧脚本不直接改写，只追加新脚本
4. 生产或答辩环境执行前先备份数据库
5. 影响核心交易表的变更必须同步更新相关文档和测试用例

不建议在课程设计阶段引入复杂在线迁移框架，保持脚本直观、可复核即可。
