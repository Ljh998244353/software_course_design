## 4. 主流程闭环

本次联调按如下链路完成：

1. 管理员登录，卖家/买家注册并登录。
2. 卖家创建拍品草稿、上传图片并提交审核。
3. 管理员审核通过拍品。
4. 管理员创建拍卖活动，调度器把活动从 `PENDING_START` 推进到 `RUNNING`。
5. 两个买家依次出价，系统完成最高价更新、竞价历史写入、广播推送和超越提醒。
6. 调度器结束拍卖，把活动从 `RUNNING` 推进到 `SETTLING`。
7. 订单调度扫描 `SETTLING` 活动并生成唯一 `PENDING_PAYMENT` 订单。
8. 获胜买家发起支付，支付回调验签成功后把订单推进到 `PAID`，并把活动/拍品推进到 `SOLD`。
9. 卖家确认发货，把订单从 `PAID` 推进到 `SHIPPED`。
10. 买家确认收货，把订单从 `SHIPPED` 推进到 `COMPLETED`。
11. 买卖双方互评，第二条评价提交后把订单从 `COMPLETED` 推进到 `REVIEWED`。
12. 管理员重建当日统计，客服查询操作日志和任务日志，确认全链路可观测。

## 5. 关键状态结果

本次联调确认以下状态切换已经闭环：

| 业务对象 | 状态流转 |
|---|---|
| `item` | `DRAFT -> PENDING_AUDIT -> READY_FOR_AUCTION -> IN_AUCTION -> SOLD` |
| `auction` | `PENDING_START -> RUNNING -> SETTLING -> SOLD` |
| `order_info` | `PENDING_PAYMENT -> PAID -> SHIPPED -> COMPLETED -> REVIEWED` |
| `payment_record` | `WAITING_CALLBACK -> SUCCESS` |
| `notification.push_status` | `PENDING -> SENT` |

同时确认以下旁路行为成立：

- 出价成功后广播 `PRICE_UPDATED`
- 被超越用户收到 `OUTBID` 站内通知和直推事件
- 结算建单、支付成功、发货、确认收货、收到评价均会补发站内通知
- `task_log` 中可查询到 `ORDER_SETTLEMENT`
- `operation_log` 中可查询到 `payment_callback`、`ship_order`、`confirm_receipt`、`submit_review`

## 10. S17 HTTP 接入基线补充

`S17` 已在本次服务层联调结论之外，补齐了一层最小可运行的真实业务 HTTP 接入基线。该基线的目标不是一次性做完所有控制器，而是先让完整前端开发拥有稳定的入口、统一的错误返回和一个真实可用的登录链路。

本次新增的真实业务入口如下：

| 用途 | 已落地入口 |
|---|---|
| 完整前端骨架入口 | `GET /app` `GET /app/` |

| 真实业务登录接口 | `POST /api/auth/login` |
| 受保护测试接口 | `GET /api/system/context` |
| 未知业务接口统一错误 | 未注册 `/api/...` 路径返回 JSON `404` |

本次基线补齐的公共能力如下：

- 新增 `HttpServiceContext`，把认证、中间件、通知、拍卖、订单、支付、评价、统计和运维服务统一装配，供后续控制器复用。

- `GET /api/system/context` 会校验 `Authorization: Bearer <token>`，并返回当前登录用户的统一响应结构，作为后续受保护业务路由的最小基线。
- `POST /api/auth/login` 当前已直接复用 `AuthService::Login()`，不再依赖只读演示接口。
