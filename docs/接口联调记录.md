## 4. 主流程闭环

本次联调按如下链路完成：

1. 管理员登录，卖家/买家注册并登录。
2. 卖家创建拍品草稿、上传图片并提交审核。
3. 管理员审核通过拍品。
4. 管理员创建拍卖活动，调度器把活动从 `PENDING_START` 推进到 `RUNNING`。
5. 两个买家依次出价，系统完成最高价更新、竞价历史写入、广播推送和超越提醒。
6. 调度器结束拍卖，把活动从 `RUNNING` 推进到 `SETTLING`。
7. 订单调度扫描 `SETTLING` 活动并生成唯一 `PENDING_PAYMENT` 订单。
8. 获胜买家发起支付，支付回调验签成功后把订单推进到 `PAID`，并把活动/拍品推进到 `SOLD`。
9. 卖家确认发货，把订单从 `PAID` 推进到 `SHIPPED`。
10. 买家确认收货，把订单从 `SHIPPED` 推进到 `COMPLETED`。
11. 买卖双方互评，第二条评价提交后把订单从 `COMPLETED` 推进到 `REVIEWED`。
12. 管理员重建当日统计，客服查询操作日志和任务日志，确认全链路可观测。

## 5. 关键状态结果

本次联调确认以下状态切换已经闭环：

| 业务对象 | 状态流转 |
|---|---|
| `item` | `DRAFT -> PENDING_AUDIT -> READY_FOR_AUCTION -> IN_AUCTION -> SOLD` |
| `auction` | `PENDING_START -> RUNNING -> SETTLING -> SOLD` |
| `order_info` | `PENDING_PAYMENT -> PAID -> SHIPPED -> COMPLETED -> REVIEWED` |
| `payment_record` | `WAITING_CALLBACK -> SUCCESS` |
| `notification.push_status` | `PENDING -> SENT` |

同时确认以下旁路行为成立：

- 出价成功后广播 `PRICE_UPDATED`
- 被超越用户收到 `OUTBID` 站内通知和直推事件
- 结算建单、支付成功、发货、确认收货、收到评价均会补发站内通知
- `task_log` 中可查询到 `ORDER_SETTLEMENT`
- `operation_log` 中可查询到 `payment_callback`、`ship_order`、`confirm_receipt`、`submit_review`

## 10. S17 HTTP 接入基线补充

`S17` 已在本次服务层联调结论之外，补齐了一层最小可运行的真实业务 HTTP 接入基线。该基线的目标不是一次性做完所有控制器，而是先让完整前端开发拥有稳定的入口、统一的错误返回和一个真实可用的登录链路。

本次新增的真实业务入口如下：

| 用途 | 已落地入口 |
|---|---|
| 完整前端骨架入口 | `GET /app` `GET /app/` |
| 完整前端静态资源 | `GET /assets/app/app.css` `GET /assets/app/app.js` |
| 真实业务登录接口 | `POST /api/auth/login` |
| 受保护测试接口 | `GET /api/system/context` |
| 未知业务接口统一错误 | 未注册 `/api/...` 路径返回 JSON `404` |

本次基线补齐的公共能力如下：

- 新增 `HttpServiceContext`，把认证、中间件、通知、拍卖、订单、支付、评价、统计和运维服务统一装配，供后续控制器复用。
- 新增统一 JSON 请求解析、异常到错误码映射、错误码到 HTTP 状态映射，以及认证上下文缓存。
- `GET /api/system/context` 会校验 `Authorization: Bearer <token>`，并返回当前登录用户的统一响应结构，作为后续受保护业务路由的最小基线。
- `POST /api/auth/login` 当前已直接复用 `AuthService::Login()`，不再依赖只读演示接口。

`S17` 实际验证命令如下：

```bash
cmake -S . -B build
cmake --build build
scripts/test_http.sh
scripts/test.sh http
ctest --test-dir build --output-on-failure
```

验证结论如下：

- `/app`、`/app/` 页面可打开，`/assets/app/app.css`、`/assets/app/app.js` 返回 `200`
- `POST /api/auth/login` 可返回统一登录结果
- `GET /api/system/context` 在带 Token 时返回统一成功响应，缺少 Token 时返回统一 `401`
- 未注册 `/api/not-found` 返回统一 JSON `404`
- 全量 `ctest` 更新为 `17/17` 全部通过

当前边界仍然如下：

- `S17` 只补最小业务 HTTP 地基，不等于 `S18` 的完整认证前端，也不等于后续拍品、拍卖、出价、订单支付页面已经完成。
- 除 `POST /api/auth/login`、`GET /api/system/context` 和演示只读接口外，其他真实业务 HTTP 控制器仍待后续步骤逐步接入。

## 11. S18 认证、会话和角色路由补充

`S18` 已在 `S17` 基线上补齐认证模块真实 HTTP 接入和 `/app` 前端登录态闭环。

本次新增和扩展的真实业务入口如下：

| 用途 | 已落地入口 |
|---|---|
| 用户注册 | `POST /api/auth/register` |
| 用户登录 | `POST /api/auth/login` |
| 用户登出 | `POST /api/auth/logout` |
| 当前用户 | `GET /api/auth/me` |
| 管理员修改用户状态 | `PATCH /api/admin/users/{id}/status` |
| 角色路由前端 | `GET /app` `GET /app/` |

本次前端补齐能力如下：

- `/app` 已从最小 HTTP 控制台升级为认证工作台。
- 页面使用 `sessionStorage` 保存 Token，刷新后调用 `GET /api/auth/me` 恢复会话。
- `USER` 可看到买家、卖家入口，`ADMIN` 可看到管理入口，`SUPPORT` 可看到运维入口。
- 未登录访问受保护 hash 路由时会回到登录/注册视图。
- 管理员页面可调用真实状态变更接口；`SUPPORT` 调用同一接口会被统一 `403` 拒绝。

`S18` 实际验证命令如下：

```bash

scripts/test_http.sh
scripts/test.sh http
```

验证结论如下：

- 注册普通用户成功，返回 `USER` 和 `ACTIVE`
- 注册用户可登录并通过 `GET /api/auth/me` 恢复当前用户
- 登出后原 Token 访问当前用户接口返回统一 `401`
- `SUPPORT` 修改用户状态返回统一 `403`
- `ADMIN` 修改用户状态成功，并能让被冻结用户后续登录返回统一 `401`
- `/app`、`/app/`、CSS、JS、`GET /api/system/context` 和统一 JSON `404` 仍保持通过
