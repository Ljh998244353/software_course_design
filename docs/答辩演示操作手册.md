# 在线拍卖平台答辩演示操作手册

## 1. 演示目标

本手册用于现场答辩时快速复现在线拍卖平台功能。当前推荐演示方式是“真实业务工作台 + 数据库事实 + 自动化测试证据”；历史上的 `/demo` 只读演示台仅作为旧交付形态说明。

演示重点：

- 系统能启动并提供浏览器入口。
- 页面数据来自 MySQL 本地数据源，不是静态截图。
- 当前主展示入口优先为 `http://127.0.0.1:3000` 前端开发服务，使用真实 `/api/...` 接口做业务操作展示。
- 主流程覆盖拍品、审核、拍卖、出价、订单、支付、评价、统计和运维。
- 高风险链路有自动化测试证明。

## 5. 推荐讲解顺序

### 5.1 需求和架构

先说明项目背景：

- 面向校园或小型社区的在线拍卖平台。
- 覆盖发布、审核、竞价、结算、评价、统计和运维。
- 后端采用 `C++20 + Drogon + MySQL` 的模块化单体。

再说明关键架构约束：

- MySQL 是拍卖、出价、订单、支付等交易事实的唯一权威数据源。
- Redis 或进程内缓存只做热点读取和旁路优化，不决定最终一致性。
- 通知和统计不阻塞出价、订单、支付等核心链路。

### 5.2 浏览器主流程

打开前端首页，按角色登录展示真实业务入口：

1. 管理员：展示待审拍品、审核日志、拍卖创建/编辑/取消、统计日报和异常处理。
2. 卖家：展示拍品草稿、图片元数据、提交审核、订单发货和评价入口。
3. 买家：展示公开拍卖、当前价、出价历史、提交出价、支付、确认收货和评价入口。
4. 客服/运维：展示操作日志、任务日志和异常看板。

### 5.3 数据库事实证明

也可以直接查询 MySQL：

```bash
mysql --socket=build/test_mysql/mysql-runtime.sock -uroot auction_system
```

注意：当前测试库名可能因脚本运行方式不同而变化，应以 `scripts/db/setup_local_mysql.sh` 或当前配置实际输出为准。

进入 MySQL 后执行：

```sql
SELECT i.title, i.item_status, a.status, a.current_price
FROM item i
JOIN auction a ON a.item_id = i.item_id
WHERE i.title LIKE '演示%';

SELECT order_no, final_amount, order_status, paid_at, completed_at
FROM order_info
WHERE order_no LIKE 'DEMO-%';
```

讲解重点是：页面上的拍卖状态、最高价和订单状态都能在数据库中对应。

## 7. 常见问题处理

### 7.1 浏览器出现 404

如果访问的是：

```text
http://127.0.0.1:18080/api/items
```

该地址直接用浏览器访问时是 `GET /api/items`，当前只注册了 `POST /api/items` 创建拍品和 `GET /api/items/mine` 我的拍品列表，因此直接打开可能返回统一 JSON `404`。

答辩时真实业务操作应优先访问：

```text
http://127.0.0.1:3000
```

### 7.2 浏览器无法连接

如果浏览器提示连接失败，说明服务没有启动或端口不对。重新执行：

```bash
./start.sh --config config/app.local.json
```

再检查：

```bash
curl -i http://127.0.0.1:18080/healthz
```

### 7.3 页面打开但没有数据

若需要重新准备本地数据库与配置：

```bash
./start.sh --config config/app.local.json --backend-only
```

然后重启服务并刷新页面。
