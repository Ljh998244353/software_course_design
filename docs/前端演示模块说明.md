# 在线拍卖平台前端演示模块说明

## 1. 文档目的

本文档说明当前仓库中的两个浏览器入口：

- `/demo`：答辩用只读演示台
- `/app`：完整前端接入真实业务后端的业务工作台

其中 `/demo` 用于课程设计答辩时展示核心业务闭环，`/app` 从 `S18` 起用于真实业务前端，当前已接入注册、登录、登出、会话保持、角色导航、路由守卫和 `S19` 卖家拍品管理。

## 2. 功能范围

当前前端相关功能如下：

- 浏览器入口：`GET /demo`
- 首页入口：`GET /`
- 静态资源：`GET /assets/demo/app.css`、`GET /assets/demo/app.js`
- 只读演示数据接口：`GET /api/demo/dashboard`
- 前端专项测试入口：`scripts/test.sh frontend`
- 完整前端业务工作台入口：`GET /app`
- 完整前端静态资源：`GET /assets/app/app.css`、`GET /assets/app/app.js`
- 真实业务注册接口：`POST /api/auth/register`
- 真实业务登录接口：`POST /api/auth/login`
- 真实业务登出接口：`POST /api/auth/logout`
- 当前用户接口：`GET /api/auth/me`
- 管理员用户状态接口：`PATCH /api/admin/users/{id}/status`
- 受保护测试接口：`GET /api/system/context`
- 卖家拍品创建接口：`POST /api/items`
- 卖家拍品编辑接口：`PUT /api/items/{id}`
- 卖家图片元数据接口：`POST /api/items/{id}/images`
- 卖家删除图片元数据接口：`DELETE /api/items/{itemId}/images/{imageId}`
- 卖家提交审核接口：`POST /api/items/{id}/submit-audit`
- 我的拍品列表接口：`GET /api/items/mine`
- 拍品详情接口：`GET /api/items/{id}`
- HTTP 基线测试入口：`scripts/test.sh http`

前端页面覆盖以下演示内容：

- 主业务流程：拍品发布、审核、拍卖创建、出价、结算、支付、履约、评价。
- 角色说明：管理员、运维/客服、卖家、买家、竞争买家。
- 拍卖状态：已成交、竞拍中、已流拍、待开始。
- 交易闭环：订单、支付记录、双方评价。
- 高风险链路：并发出价、拍卖结束竞争、支付回调幂等、缓存降级、安全负向。
- 运维可观测：站内通知、任务日志、操作日志、统计日报。

## 4. 接口说明

### 4.1 `GET /demo`

返回前端演示页面。

页面会自动调用 `/api/demo/dashboard` 加载演示数据。如果接口失败，页面会提示先执行演示环境初始化脚本。

### 4.2 `GET /app`

返回完整前端业务工作台。

该页面当前承担 `S18` 的认证与角色路由职责，以及 `S19` 的卖家拍品管理职责：

- 调用 `POST /api/auth/register`
- 调用 `POST /api/auth/login`
- 调用 `POST /api/auth/logout`
- 调用 `GET /api/auth/me` 做会话恢复
- 管理员调用 `PATCH /api/admin/users/{id}/status`
- 调用 `GET /api/system/context`
- 卖家调用 `POST /api/items` 创建拍品草稿
- 卖家调用 `PUT /api/items/{id}` 编辑自己的草稿或被驳回拍品
- 卖家调用 `POST /api/items/{id}/images` 写入图片元数据
- 卖家调用 `DELETE /api/items/{itemId}/images/{imageId}` 删除图片元数据
- 卖家调用 `GET /api/items/mine` 和 `GET /api/items/{id}` 查看自己的拍品
- 卖家调用 `POST /api/items/{id}/submit-audit` 提交审核
- 根据 `USER`、`ADMIN`、`SUPPORT` 显示不同导航入口
- 未登录访问受保护 hash 路由时回到登录/注册视图
- 展示统一响应 JSON，并验证未知 `/api/...` 路径是否返回统一 JSON `404`

它仍不是完整业务前端，管理员审核与建拍、拍卖列表、出价、订单支付、评价、统计运维等页面将在 `S20-S25` 继续接入。

### 4.3 `POST /api/auth/register`

请求体示例：

```json
{
  "username": "seller_new",
  "password": "UserPass1",
  "phone": "13900000000",
  "email": "seller_new@auction.local",
  "nickname": "新卖家"
}
```

返回统一响应结构，`data` 中包含：

| 字段 | 说明 |
|---|---|
| `userId` | 新注册用户 ID |
| `username` | 用户名 |
| `roleCode` | 固定为 `USER` |
| `status` | 固定为 `ACTIVE` |

### 4.4 `POST /api/auth/login`

请求体示例：

```json
{
  "username": "admin",
  "password": "Admin@123"
}
```

返回统一响应结构，`data` 中包含：

| 字段 | 说明 |
|---|---|
| `token` | 登录 Token |
| `expireAt` | 过期时间 |
| `userInfo.userId` | 用户 ID |
| `userInfo.username` | 用户名 |
| `userInfo.nickname` | 昵称 |
| `userInfo.roleCode` | 角色 |
| `userInfo.status` | 账号状态 |

### 4.5 `POST /api/auth/logout`

请求头：

```text
Authorization: Bearer <token>
```

成功返回：

```json
{
  "code": 0,
  "message": "success",
  "data": {
    "success": true
  }
}
```

登出后同一 Token 再访问受保护接口会返回统一 `401`。

### 4.6 `GET /api/auth/me`

用于 `/app` 初始化时恢复登录态。

请求头：

```text
Authorization: Bearer <token>
```

成功返回的 `data` 中包含 `userId`、`username`、`nickname`、`roleCode`、`status` 和 `tokenExpireAt`。

### 4.7 `PATCH /api/admin/users/{id}/status`

用途：

- 管理员修改目标用户账号状态，并触发目标用户已有会话失效。

鉴权要求：

- 仅 `ADMIN`

请求体示例：

```json
{
  "status": "FROZEN",
  "reason": "risk control"
}
```

成功返回的 `data` 中包含 `userId`、`oldStatus`、`newStatus` 和 `updatedAt`。`SUPPORT` 或普通用户调用会返回统一 `403`。

### 4.8 `GET /api/system/context`

用途：

- 作为 `S17` 的受保护测试接口，验证认证上下文、统一错误映射和后续控制器复用方式。

请求头：

```text
Authorization: Bearer <token>
```

成功返回的 `data` 中包含：

| 字段 | 说明 |
|---|---|
| `view` | 固定为 `business_http_baseline` |
| `authenticated` | 当前是否已通过认证 |
| `requestPath` | 当前请求路径 |
| `tokenExpireAt` | Token 过期时间 |
| `user` | 当前登录用户的认证上下文 |

缺少 Token 时会返回统一 JSON `401`；未知 `/api/...` 路径会返回统一 JSON `404`。

### 4.9 S19 卖家拍品接口

`/app` 的卖家视图已接入以下真实业务接口：

| 接口 | 用途 |
|---|---|
| `POST /api/items` | 创建拍品草稿 |
| `PUT /api/items/{id}` | 编辑草稿或被驳回拍品 |
| `POST /api/items/{id}/images` | 添加图片元数据 |
| `DELETE /api/items/{itemId}/images/{imageId}` | 删除图片元数据 |
| `GET /api/items/mine` | 查询我的拍品列表 |
| `GET /api/items/{id}` | 查询拍品详情和最近审核结果 |
| `POST /api/items/{id}/submit-audit` | 提交审核 |

当前图片入口以 `imageUrl`、`sortNo`、`isCover` 保存图片元数据，不做 multipart 文件上传。

### 4.10 `GET /api/demo/dashboard`

返回统一响应结构：

```json
{
  "code": 0,
  "message": "success",
  "data": {
    "view": "demo_dashboard",
    "mode": "read_only_demo",
    "summary": {},
    "accounts": [],
    "auctions": [],
    "bids": [],
    "orders": [],
    "payments": [],
    "reviews": [],
    "notifications": [],
    "taskLogs": [],
    "operationLogs": [],
    "statisticsDaily": [],
    "riskScenarios": []
  }
}
```

主要字段说明：

| 字段 | 说明 |
|---|---|
| `summary` | 演示账号数、拍品数、拍卖数、出价数、订单数和成交金额 |
| `accounts` | 演示账号、角色、状态和答辩用途 |
| `auctions` | 拍卖活动、当前最高价、最高出价者、延时保护配置和状态 |
| `bids` | 最近出价记录 |
| `orders` | 演示成交订单 |
| `payments` | mock 支付记录 |
| `reviews` | 买卖双方评价 |
| `notifications` | 站内通知 |
| `taskLogs` | 调度任务日志 |
| `operationLogs` | 管理操作日志 |
| `statisticsDaily` | 统计日报 |
| `riskScenarios` | 高风险测试说明卡片 |

## 5. 数据来源

本模块只读取 MySQL，不写入交易状态。默认数据来源如下：

- 数据库：`auction_demo`
- 初始化脚本：`scripts/deploy/init_demo_env.sh`
- 演示数据：`sql/demo_data.sql`

这种设计保证演示页面展示的内容来自数据库事实，同时避免前端演示误改核心拍卖、订单和支付状态。

## 6. 运行方式

初始化演示环境：

```bash
./scripts/deploy/init_demo_env.sh
```

启动服务：

```bash
./scripts/deploy/run_demo_server.sh
```

浏览器打开：

```text
http://127.0.0.1:18080/demo
http://127.0.0.1:18080/app
```

接口验证：

```bash
curl http://127.0.0.1:18080/api/demo/dashboard
```

## 7. 验证方式

前端演示数据专项验证：

```bash
scripts/test.sh frontend
scripts/test.sh http
```

该入口会：

- 构建工程。
- 初始化 `auction_demo`。
- 导入 `sql/demo_data.sql`。
- 执行 `auction_demo_dashboard_tests`，校验演示接口返回账号、拍品、拍卖、订单、支付和高风险说明。
- 执行 `auction_http_baseline`，校验 `/app`、`/app/`、`/assets/app/app.css`、`/assets/app/app.js`、注册、登录、`/api/auth/me`、登出、管理员用户状态、越权拒绝、`/api/system/context` 和统一 JSON `404`。
- `auction_http_baseline` 同时覆盖卖家拍品创建、非法字段、图片元数据、删除图片、编辑、列表、详情、提交审核和提审后禁止编辑。

全量验证仍使用：

```bash
ctest --test-dir build --output-on-failure
```

## 8. 边界说明

- `/demo` 是只读演示台，不提供真实出价、支付、审核等写操作入口。
- `/app` 已完成认证、会话、角色路由和卖家拍品管理，不代表管理员审核建拍、拍卖大厅、出价、订单支付等业务页面已经完成。
- 核心交易写路径仍以服务层实现和自动化测试为准。
- 当前已接入的真实业务 HTTP 包括认证、用户状态和卖家拍品接口；审核、拍卖、出价、订单支付、评价、统计运维等控制器将在 `S20-S30` 继续补齐。
- 未知 `/api/...` 路径现在会返回统一 JSON `404`，不再直接返回裸 `404 Not Found` 页面。
- 若 `/demo` 页面能打开但数据加载失败，优先检查是否已执行 `scripts/deploy/init_demo_env.sh`，以及演示服务是否使用 `build/demo_config/app.demo.json` 启动。
