# 在线拍卖平台前端演示模块说明

## 1. 文档目的

本文档说明当前仓库中的两个浏览器入口：

- `/demo`：答辩用只读演示台
- `/app`：完整前端接入真实业务后端的业务工作台

其中 `/demo` 用于课程设计答辩时展示核心业务闭环，`/app` 从 `S18` 起用于真实业务前端，当前已接入注册、登录、登出、会话保持、角色导航、路由守卫、`S19` 卖家拍品管理、`S20` 管理员审核建拍、`S21` 买家拍卖大厅、`S22` 真实出价与通知轮询、`S23` 订单支付和履约页面、`S24` 评价、通知中心和用户信用展示、`S25` 统计报表和运维异常页面，并在 `S26` 完成全站 UX、状态处理和安全前端约束收口。

## 4. 接口说明

### 4.1 `GET /demo`

返回前端演示页面。

页面会自动调用 `/api/demo/dashboard` 加载演示数据。如果接口失败，页面会提示先执行演示环境初始化脚本。

### 4.2 `GET /app`

返回完整前端业务工作台。

该页面当前承担 `S18` 的认证与角色路由职责、`S19` 的卖家拍品管理职责、`S20` 的管理员审核建拍职责、`S21` 的买家拍卖大厅职责、`S22` 的真实出价和通知轮询职责、`S23` 的订单支付和履约职责、`S24` 的评价、通知中心和信用展示职责、`S25` 的统计报表和运维异常页面职责，以及 `S26` 的全站 UX、状态处理和安全前端约束职责：

- 调用 `POST /api/auth/register`
- 调用 `POST /api/auth/login`
- 调用 `POST /api/auth/logout`
- 调用 `GET /api/auth/me` 做会话恢复
- 管理员调用 `PATCH /api/admin/users/{id}/status`
- 调用 `GET /api/system/context`
- 卖家调用 `POST /api/items` 创建拍品草稿
- 卖家调用 `PUT /api/items/{id}` 编辑自己的草稿或被驳回拍品
- 卖家调用 `POST /api/items/{id}/images` 写入图片元数据
- 卖家调用 `DELETE /api/items/{itemId}/images/{imageId}` 删除图片元数据
- 卖家调用 `GET /api/items/mine` 和 `GET /api/items/{id}` 查看自己的拍品
- 卖家调用 `POST /api/items/{id}/submit-audit` 提交审核
- 管理员调用 `GET /api/admin/items/pending` 查看待审核拍品
- 管理员调用 `POST /api/admin/items/{id}/audit` 审核通过或驳回
- 管理员调用 `GET /api/admin/items/{id}/audit-logs` 查看审核日志
- 管理员调用 `POST /api/admin/auctions` 创建拍卖
- 管理员调用 `GET /api/admin/auctions` 和 `GET /api/admin/auctions/{id}` 管理拍卖列表和详情
- 管理员调用 `PUT /api/admin/auctions/{id}` 编辑待开始拍卖
- 管理员调用 `POST /api/admin/auctions/{id}/cancel` 取消待开始拍卖
- 买家调用 `GET /api/auctions` 浏览公开拍卖列表
- 买家调用 `GET /api/auctions/{id}` 查看公开拍卖详情
- 买家调用 `GET /api/auctions/{id}/price` 查看当前最高价和下一口价格
- 买家调用 `GET /api/auctions/{id}/bids` 查看出价历史
- 买家调用 `POST /api/auctions/{id}/bids` 提交真实出价
- 买家调用 `GET /api/auctions/{id}/my-bid` 查看自己是否仍是当前最高出价者
- 登录用户调用 `GET /api/notifications` 轮询站内通知
- 登录用户调用 `PATCH /api/notifications/{id}/read` 标记通知已读
- 登录用户调用 `GET /api/orders/mine` 查询自己作为买家或卖家的订单
- 登录用户调用 `GET /api/orders/{id}` 查看订单详情
- 买家调用 `POST /api/orders/{id}/pay` 发起 mock 支付
- 登录用户调用 `GET /api/orders/{id}/payment` 查询支付状态
- mock 支付服务调用 `POST /api/payments/callback` 推进支付结果
- 卖家调用 `POST /api/orders/{id}/ship` 确认发货
- 买家调用 `POST /api/orders/{id}/confirm-receipt` 确认收货
- 买卖双方调用 `POST /api/reviews` 对已完成订单提交评价
- 登录用户调用 `GET /api/orders/{id}/reviews` 查看本人参与订单的互评记录
- 登录用户调用 `GET /api/users/{id}/reviews/summary` 查看交易对象信用摘要
- 登录用户进入通知中心调用 `GET /api/notifications` 查看站内通知
- 登录用户在通知中心调用 `PATCH /api/notifications/{id}/read` 标记通知已读
- 管理员调用 `GET /api/admin/statistics/daily` 查询日报
- 管理员调用 `POST /api/admin/statistics/daily/rebuild` 重跑指定日期日报
- 管理员调用 `GET /api/admin/statistics/daily/export` 导出 CSV 内容
- 管理员或客服调用 `GET /api/admin/ops/operation-logs` 查询操作日志
- 管理员或客服调用 `GET /api/admin/ops/task-logs` 查询任务日志
- 管理员或客服调用 `GET /api/admin/ops/exceptions` 查询异常看板
- 管理员或客服调用 `POST /api/admin/ops/exceptions/mark` 手工标记异常
- 管理员调用 `POST /api/admin/ops/notifications/retry` 重试失败通知
- 管理员调用 `POST /api/admin/ops/compensations` 执行补偿入口
- 根据 `USER`、`ADMIN`、`SUPPORT` 显示不同导航入口
- 未登录访问受保护 hash 路由时回到登录/注册视图，已登录但角色无权访问时回退到该角色可见入口
- 关键表单使用 `required`、`maxlength`、`minlength`、`pattern`、`min/max` 和 JS 校验限制，金额、日期、数量和拍卖时间在提交前先做前端反馈
- 全局请求状态同步到顶部和 API 区状态徽标，请求进行中会设置 `body.is-busy` 与 `aria-busy`
- 管理员专属运维写操作在非 `ADMIN` 角色下禁用控件并显示“仅管理员”，后端仍保留最终鉴权
- 展示统一响应 JSON，并验证未知 `/api/...` 路径是否返回统一 JSON `404`

它仍不是最终业务前端，真实 HTTP 回归收口、页面级验收和部署脚本将在 `S27-S30` 继续接入。

### 4.3 `POST /api/auth/register`

请求体示例：

```json
{
  "username": "seller_new",
  "password": "UserPass1",
  "phone": "13900000000",
  "email": "seller_new@auction.local",
  "nickname": "新卖家"
}
```

返回统一响应结构，`data` 中包含：

| 字段 | 说明 |
|---|---|
| `userId` | 新注册用户 ID |
| `username` | 用户名 |
| `roleCode` | 固定为 `USER` |
| `status` | 固定为 `ACTIVE` |

### 4.4 `POST /api/auth/login`

请求体示例：

```json
{
  "username": "admin",
  "password": "Admin@123"
}
```

返回统一响应结构，`data` 中包含：

| 字段 | 说明 |
|---|---|
| `token` | 登录 Token |
| `expireAt` | 过期时间 |
| `userInfo.userId` | 用户 ID |
| `userInfo.username` | 用户名 |
| `userInfo.nickname` | 昵称 |
| `userInfo.roleCode` | 角色 |
| `userInfo.status` | 账号状态 |

### 4.5 `POST /api/auth/logout`

请求头：

```text
Authorization: Bearer <token>
```

成功返回：

```json
{
  "code": 0,
  "message": "success",
  "data": {
    "success": true
  }
}
```

登出后同一 Token 再访问受保护接口会返回统一 `401`。

### 4.6 `GET /api/auth/me`

用于 `/app` 初始化时恢复登录态。

请求头：

```text
Authorization: Bearer <token>
```

成功返回的 `data` 中包含 `userId`、`username`、`nickname`、`roleCode`、`status` 和 `tokenExpireAt`。

### 4.7 `PATCH /api/admin/users/{id}/status`

用途：

- 管理员修改目标用户账号状态，并触发目标用户已有会话失效。

鉴权要求：

- 仅 `ADMIN`

请求体示例：

```json
{
  "status": "FROZEN",
  "reason": "risk control"
}
```

成功返回的 `data` 中包含 `userId`、`oldStatus`、`newStatus` 和 `updatedAt`。`SUPPORT` 或普通用户调用会返回统一 `403`。

### 4.8 `GET /api/system/context`

用途：

- 作为 `S17` 的受保护测试接口，验证认证上下文、统一错误映射和后续控制器复用方式。

请求头：

```text
Authorization: Bearer <token>
```

成功返回的 `data` 中包含：

| 字段 | 说明 |
|---|---|
| `view` | 固定为 `business_http_baseline` |
| `authenticated` | 当前是否已通过认证 |
| `requestPath` | 当前请求路径 |
| `tokenExpireAt` | Token 过期时间 |
| `user` | 当前登录用户的认证上下文 |

缺少 Token 时会返回统一 JSON `401`；未知 `/api/...` 路径会返回统一 JSON `404`。

### 4.12 S22 真实出价与通知接口

`/app` 的买家视图已接入以下真实业务接口：

| 接口 | 用途 |
|---|---|
| `POST /api/auctions/{id}/bids` | 提交真实出价，前端每次生成唯一 `requestId` |
| `GET /api/auctions/{id}/my-bid` | 查询当前用户在该活动中的最高出价和领先状态 |

当前页面支持在 `RUNNING` 活动详情内输入金额并提交出价；出价成功后刷新当前价、下一口、出价历史和个人领先状态。页面每 10 秒轮询当前详情的价格、出价历史、个人出价状态和站内通知，用于展示被超越提示。低价、过期、冻结会话和幂等冲突均通过统一响应区与 toast 反馈。

## 5. 数据来源

本模块只读取 MySQL，不写入交易状态。默认数据来源如下：

- 数据库：`auction_demo`
- 初始化脚本：`scripts/deploy/init_demo_env.sh`
- 演示数据：`sql/demo_data.sql`

这种设计保证演示页面展示的内容来自数据库事实，同时避免前端演示误改核心拍卖、订单和支付状态。

## 8. 边界说明

- `/demo` 是只读演示台，不提供真实出价、支付、审核等写操作入口。
- `/app` 已完成认证、会话、角色路由、卖家拍品管理、管理员审核建拍、买家拍卖大厅、真实出价、通知轮询、订单支付、履约、评价、通知中心、信用展示、统计报表、运维异常页面、全站 UX、状态处理和安全前端约束，但还没有完成页面级自动化验收。
- 核心交易写路径仍以服务层实现和自动化测试为准。
- 当前已接入的真实业务 HTTP 包括认证、用户状态、卖家拍品、管理员审核、管理端拍卖、公开拍卖、出价、通知轮询、订单支付、履约、评价、信用、统计和运维接口；后续 `S27-S30` 将继续收口回归脚本、页面验收、部署和最终 handoff。
- 未知 `/api/...` 路径现在会返回统一 JSON `404`，不再直接返回裸 `404 Not Found` 页面。
- 若 `/demo` 页面能打开但数据加载失败，优先检查是否已执行 `scripts/deploy/init_demo_env.sh`，以及演示服务是否使用 `build/demo_config/app.demo.json` 启动。
