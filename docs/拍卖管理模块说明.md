# 在线拍卖平台拍卖管理模块说明

## 1. 文档目的

本文档用于完成 `S06` 和 `S20` 前端真实接入，明确拍卖管理模块在课程设计阶段的职责范围、业务流程、状态流转、接口契约、关键错误码、事务边界和依赖关系。

本模块位于物品审核模块之后、竞价模块之前，负责把已审核通过的拍品配置为可执行的拍卖活动，并提供按时间自动切换状态的基础能力。

## 2. 模块职责

拍卖管理模块负责以下能力：

- 管理员基于审核通过的拍品创建拍卖活动
- 管理员维护拍卖活动的时间规则、起拍价、加价幅度和延时保护参数
- 管理员查看拍卖活动列表、详情和当前状态
- 系统按开始时间自动启动活动
- 系统按结束时间自动结束活动，并根据是否存在有效最高出价推进到流拍或待结算状态
- 为后续竞价、订单和通知模块提供统一的活动状态基础

本模块不负责以下内容：

- 不负责具体出价合法性校验和最高价更新
- 不负责写入竞价历史
- 不负责直接生成订单或处理支付
- 不负责最终评价、统计和报表口径计算

## 3. 核心实体与依赖

### 3.1 核心实体

本模块核心实体如下：

- `Auction`
- `Item`
- `TaskLog`

### 3.2 主要依赖

| 依赖对象 | 作用 | 边界说明 |
|---|---|---|
| 认证与权限模块 | 识别当前用户身份、角色和账号状态 | 所有管理写接口必须先完成登录校验和管理员权限校验 |
| `item` 表 | 提供可拍卖拍品和拍品状态 | 仅 `READY_FOR_AUCTION` 的拍品允许创建活动 |
| `auction` 表 | 存储活动主状态、时间规则和价格基线 | 是活动状态的权威来源 |
| `task_log` 表 | 记录开始、结束、补偿等调度执行情况 | 调度失败要可追踪、可重试 |
| 竞价模块 | 读取活动状态并在有效竞拍期内写入出价 | 拍卖管理模块不负责最高价竞争逻辑 |
| 订单模块 | 处理 `SETTLING` 状态的活动并生成订单 | 本模块只负责把有最高出价的活动推进到待结算 |
| 通知模块 | 发送活动开始、结束、取消等通知 | 通知失败不回滚主事务 |

## 4. 角色与权限边界

### 4.1 管理员权限

`ADMIN` 在拍卖管理模块中具备以下能力：

- 创建拍卖活动
- 修改待开始活动
- 取消待开始活动
- 查看全部活动列表、详情和状态
- 触发人工补偿或重试调度任务

`ADMIN` 不负责：

- 直接修改竞价历史
- 绕过状态机强行生成订单
- 修改已结束活动的成交事实

### 4.2 普通用户权限

`USER` 在本模块中具备以下只读能力：

- 查看公共拍卖活动列表
- 查看拍卖活动详情、时间规则和当前状态
- 作为卖家查看自己拍品对应的活动进度

`USER` 不允许：

- 创建、修改或取消拍卖活动
- 手工切换活动状态

### 4.3 客服/运维权限

`SUPPORT` 默认具备只读查询能力：

- 查看活动详情
- 查看调度任务结果
- 查询异常活动和补偿记录

`SUPPORT` 不允许：

- 创建、修改或取消拍卖活动
- 手工变更拍卖状态

## 5. 状态模型

### 5.1 拍卖活动状态定义

`auction.status` 建议采用以下状态：

- `PENDING_START`：待开始，活动已创建但尚未到开始时间
- `RUNNING`：竞拍中，允许竞价模块接受有效出价
- `SETTLING`：已结束待结算，存在有效最高出价，等待订单模块生成订单并推进结果
- `SOLD`：已成交，结算完成
- `UNSOLD`：已流拍，无有效出价或结算未成交
- `CANCELLED`：已取消，仅允许在开始前取消

### 5.2 状态流转规则

```text
PENDING_START -> RUNNING
PENDING_START -> CANCELLED
RUNNING -> SETTLING
RUNNING -> UNSOLD
SETTLING -> SOLD
SETTLING -> UNSOLD
```

状态说明如下：

- 活动创建成功后默认为 `PENDING_START`
- 到达开始时间后由调度任务切换为 `RUNNING`
- 到达结束时间后，若无有效最高出价则直接转为 `UNSOLD`
- 到达结束时间后，若存在有效最高出价则转为 `SETTLING`
- `SETTLING` 由订单模块在后续步骤中回写 `SOLD` 或 `UNSOLD`
- `CANCELLED`、`SOLD`、`UNSOLD` 为终态

### 5.3 与拍品状态的协同规则

`auction` 与 `item` 的状态协同约定如下：

- 仅 `item.item_status = READY_FOR_AUCTION` 的拍品允许创建活动
- 创建活动成功时，拍品状态保持 `READY_FOR_AUCTION`，等待真正开拍
- 活动开始时，调度任务在同一事务中把拍品状态从 `READY_FOR_AUCTION` 更新为 `IN_AUCTION`
- 活动结束且无人出价时，调度任务在同一事务中把拍品状态从 `IN_AUCTION` 更新为 `UNSOLD`
- 活动结束且存在最高出价时，活动进入 `SETTLING`，由于当前拍品状态模型未单独定义“待结算”，拍品暂保持 `IN_AUCTION`，待后续订单结算模块回写为 `SOLD` 或 `UNSOLD`
- 待开始活动被取消时，拍品状态保持 `READY_FOR_AUCTION`

### 5.4 时间规则

拍卖管理模块对时间参数采用以下基线规则：

- `start_time` 必须晚于活动创建时间
- `end_time` 必须晚于 `start_time`
- 延时保护关闭时，`anti_sniping_window_seconds = 0` 且 `extend_seconds = 0`
- 延时保护开启时，`anti_sniping_window_seconds > 0` 且 `extend_seconds > 0`
- 调度任务在结束活动前必须读取活动的最新 `end_time`，避免与竞价模块的延时保护冲突

## 6. 业务流程

### 6.1 管理员创建活动流程

1. 管理员选择已审核通过的拍品。
2. 系统校验当前管理员身份和账号状态。
3. 系统校验拍品存在且 `item_status = READY_FOR_AUCTION`。
4. 系统校验该拍品不存在其他非终态活动。
5. 系统校验 `start_time`、`end_time`、`start_price`、`bid_step` 和延时保护参数。
6. 系统写入 `auction` 记录，初始化 `status = PENDING_START`、`current_price = start_price`、`highest_bidder_id = null`。
7. 系统返回活动编号和待开始状态。

### 6.2 管理员修改待开始活动流程

1. 管理员对 `PENDING_START` 活动发起修改。
2. 系统锁定活动记录并校验当前状态仍为 `PENDING_START`。
3. 系统校验新的时间规则和价格规则。
4. 系统更新活动配置并记录更新时间。
5. 已进入 `RUNNING`、`SETTLING` 或终态的活动不允许修改核心规则。

### 6.3 管理员取消待开始活动流程

1. 管理员对 `PENDING_START` 活动发起取消。
2. 系统校验当前状态仍为 `PENDING_START`。
3. 系统把 `auction.status` 更新为 `CANCELLED`。
4. 拍品状态保持 `READY_FOR_AUCTION`，可后续重新配置新活动。
5. 事务提交成功后异步发送取消通知。

### 6.4 自动开始活动流程

1. 调度任务按固定周期扫描 `PENDING_START` 且 `start_time <= now` 的活动。
2. 系统锁定活动记录和关联拍品记录。
3. 系统再次校验活动状态、拍品状态和开始时间边界。
4. 系统在同一事务中把活动改为 `RUNNING`，并把拍品状态改为 `IN_AUCTION`。
5. 事务提交成功后再更新缓存并发送活动开始通知。

### 6.5 自动结束活动流程

1. 调度任务按固定周期扫描 `RUNNING` 且 `end_time <= now` 的活动。
2. 系统锁定活动记录并重新读取最新 `end_time`。
3. 若活动因延时保护已被顺延，则本次结束任务直接退出。
4. 若活动不存在 `highest_bidder_id`，则在同一事务中把活动更新为 `UNSOLD`，并把拍品更新为 `UNSOLD`。
5. 若活动存在 `highest_bidder_id`，则在同一事务中把活动更新为 `SETTLING`，并写入后续结算任务触发记录。
6. 事务提交成功后异步发送拍卖结束通知。

### 6.6 活动查询流程

1. 管理端支持按状态、拍品、卖家、时间区间查询活动。
2. 公共前台支持按关键词、状态和分页查询活动列表。
3. 详情页返回活动时间规则、起拍价、当前价、当前状态和拍品摘要信息。
4. 竞价模块和订单模块都以活动详情查询结果作为后续处理输入。

## 8. 事务边界与一致性要求

### 8.1 活动创建

事务边界：

- 锁定关联 `item` 记录
- 校验拍品状态和非终态活动冲突
- 插入 `auction` 记录

上述步骤必须在同一事务中成功或失败。

校验要求：

- 必须检查 `item_status = READY_FOR_AUCTION`
- 必须保证同一拍品同时只有一个非终态活动
- 必须校验开始和结束时间合法

### 8.2 活动修改与取消

事务边界：

- 锁定 `auction` 记录
- 校验当前状态为 `PENDING_START`
- 更新活动配置或状态

说明：

- 取消活动时不修改拍品状态
- 通知发送必须在事务提交成功后执行

### 8.3 自动开始调度

事务边界：

- 锁定 `auction` 和 `item` 记录
- 再次校验 `start_time <= now`
- 更新 `auction.status = RUNNING`
- 更新 `item.item_status = IN_AUCTION`

上述四项必须在同一事务中成功或失败。

一致性要求：

- 重复扫描时必须依赖活动当前状态保证幂等
- 若拍品状态已不满足 `READY_FOR_AUCTION`，应记录任务失败并等待人工处理

### 8.4 自动结束调度

事务边界：

- 锁定 `auction` 记录，必要时同时锁定 `item`
- 再次校验 `status = RUNNING` 且 `end_time <= now`
- 无最高出价时：更新 `auction.status = UNSOLD`，并把 `item.item_status` 更新为 `UNSOLD`
- 有最高出价时：更新 `auction.status = SETTLING`，并写入结算触发记录

上述更新必须在同一事务中成功或失败。

一致性要求：

- 结束前必须读取最新 `end_time`，避免与延时保护并发冲突
- 有最高出价的活动在 `S06` 只推进到 `SETTLING`，订单生成在后续 `S08` 完成
- 后续订单模块必须利用 `order_info.auction_id` 唯一约束避免重复生成订单

## 9. 错误码设计

拍卖管理模块建议使用 `4300-4399` 段：

| 错误码 | 说明 |
|---|---|
| 4301 | 拍卖活动不存在 |
| 4302 | 拍品不存在 |
| 4303 | 拍品状态不允许创建拍卖活动 |
| 4304 | 开始时间非法 |
| 4305 | 结束时间必须晚于开始时间 |
| 4306 | 起拍价非法 |
| 4307 | 最小加价幅度非法 |
| 4308 | 延时保护参数非法 |
| 4309 | 当前拍品已存在未结束拍卖活动 |
| 4310 | 当前状态不允许修改活动 |
| 4311 | 当前状态不允许取消活动 |
| 4312 | 当前状态不允许启动活动 |
| 4313 | 当前状态不允许结束活动 |
| 4314 | 活动状态切换冲突，请稍后重试 |
| 4315 | 关联拍品状态异常 |

同时可复用全局错误码：

- `4001`：参数非法
- `4002`：资源不存在
- `4004`：权限不足
- `5001`：内部服务异常

| `ListAdminAuctions` | 查询管理端活动列表 |

| `ListPublicAuctions` | 查询前台活动列表 |

## 11. 模块验收标准

`S06` 完成后，应满足以下标准：

- 管理员只能对 `READY_FOR_AUCTION` 拍品创建拍卖活动
- 活动创建时必须完成时间规则、价格规则和活动冲突校验
- 仅 `PENDING_START` 状态允许修改和取消
- 系统可按开始时间自动把活动切换为 `RUNNING`
- 系统可按结束时间自动把活动切换为 `UNSOLD` 或 `SETTLING`
- 活动与拍品状态协同关系明确且可追踪
- 调度任务重复执行不会造成重复状态推进
