# 在线拍卖平台认证与权限模块说明

## 1. 文档目的

本文档用于完成 `S04`，明确认证与权限模块在课程设计阶段的职责范围、业务流程、接口契约、状态模型、错误码和依赖边界。

本模块是后续物品发布、活动创建、出价、支付、评价和管理操作的统一鉴权入口。

## 2. 模块职责

认证与权限模块负责以下能力：

- 用户注册与登录
- Token 签发、解析、续期和登出
- 登录态识别与用户上下文注入
- 用户角色识别与 RBAC 权限控制
- 账号状态控制，如冻结、禁用、恢复
- 对所有写接口执行统一登录校验、角色校验和状态校验
- 对管理员敏感操作写审计日志

本模块不负责以下内容：

- 不负责物品审核业务本身
- 不负责竞价、订单、支付的业务规则判断
- 不直接决定拍卖、订单和支付状态迁移

## 3. 角色与权限模型

### 3.1 角色定义

系统采用基于角色的访问控制模型，最小角色集合如下：

- `USER`：普通用户，统一承载买家和卖家身份
- `ADMIN`：管理员，负责审核、活动配置、用户状态管理和异常处理
- `SUPPORT`：运维/客服，负责日志查询、异常协助和通知处理

说明：

- 普通用户不再拆分为独立买家表和卖家表，通过具体业务行为区分买卖角色
- 管理员与运维/客服是平台后台角色，不参与普通交易链路出价与支付

### 3.2 权限边界

| 角色 | 允许的核心能力 | 禁止的核心能力 |
|---|---|---|
| `USER` | 注册、登录、发布拍品、查看拍卖、出价、支付、评价、查看个人订单与消息 | 审核拍品、冻结其他用户、创建后台统计任务、处理异常订单 |
| `ADMIN` | 审核拍品、创建拍卖、查看统计、管理用户状态、处理异常订单 | 以普通买家身份参与竞价和支付 |
| `SUPPORT` | 查看日志、查询交易、发送通知、标记异常记录 | 审核拍品、修改用户角色、直接关闭或创建拍卖活动 |

### 3.3 权限判定原则

权限判定遵循以下规则：

- 所有需要登录的接口必须先校验 Token
- 所有写接口必须校验账号状态，`FROZEN` 和 `DISABLED` 用户不得继续执行写操作
- 管理接口必须校验 `ADMIN` 或 `SUPPORT` 角色
- 审核、活动创建、用户状态修改等接口仅允许 `ADMIN`
- 日志查询、通知发送、异常标记接口允许 `ADMIN` 和 `SUPPORT`
- 普通用户只能访问自己的个人资源，不能越权访问他人订单、评价和消息

## 4. 账号状态与会话模型

### 4.1 账号状态

`user_account.status` 建议取值如下：

- `ACTIVE`：正常可登录、可执行业务操作
- `FROZEN`：账号冻结，可拒绝登录或只允许只读访问
- `DISABLED`：账号禁用，不允许登录和继续使用系统

状态迁移建议如下：

```text
ACTIVE <-> FROZEN
ACTIVE -> DISABLED
FROZEN -> DISABLED
```

说明：

- `DISABLED` 视为终态，不建议在课程设计阶段支持恢复
- `FROZEN` 主要用于违规、风险控制或临时限制

### 4.2 Token 与登录态

课程设计阶段建议采用“签名 Token + Redis 会话缓存 + MySQL 用户状态校验”的简化方案。

Token 设计原则如下：

- 登录成功后签发访问 Token
- Token 中至少包含 `user_id`、`role_code`、`expire_at`
- 服务端对 Token 执行签名校验，防止伪造
- Redis 保存短期登录态，支持登出和失效控制
- MySQL 中的 `user_account` 仍是角色和账号状态的权威来源

建议 Redis Key 如下：

- `auth:token:{token}`：记录 `user_id`、`role_code`、`expire_at`
- `session:user:{userId}`：记录用户最近登录信息或当前会话摘要

会话处理规则如下：

- 登录成功后写入 Redis，会话 TTL 与 Token 过期时间一致
- 登出时删除 Token 对应 Redis 记录
- 请求鉴权时先校验 Token 格式与签名，再检查 Redis 会话，再读取 MySQL 用户状态
- 当 Redis 异常时，可退化为“签名 Token + MySQL 用户状态校验”的保守模式，但登出即时失效能力会下降

## 5. 业务流程

### 5.1 注册流程

1. 用户提交用户名、密码、手机号或邮箱等注册信息。
2. 系统校验字段完整性、格式合法性和唯一性。
3. 系统对密码做安全哈希后写入 `user_account`。
4. 用户初始角色为 `USER`，初始状态为 `ACTIVE`。
5. 注册成功后可引导用户直接登录。

### 5.2 登录流程

1. 用户提交用户名或手机号以及密码。
2. 系统查询 `user_account` 并校验密码哈希。
3. 系统检查账号状态是否允许登录。
4. 登录成功后生成 Token，并写入 Redis 会话缓存。
5. 返回 Token、过期时间和当前用户基础信息。

### 5.3 请求鉴权流程

1. 接入层从请求头读取 `Authorization: Bearer <token>`。
2. 认证过滤器校验 Token 格式、签名和过期时间。
3. 系统检查 Redis 中是否存在对应登录态。
4. 系统读取 MySQL 用户信息，确认用户未被冻结或禁用。
5. 系统将 `user_id`、`role_code`、`status` 注入请求上下文，供后续模块使用。

### 5.4 登出流程

1. 用户携带当前 Token 发起登出请求。
2. 系统删除 Redis 登录态。
3. 当前 Token 立即失效，后续请求应返回未登录或 Token 无效。

### 5.5 管理员修改用户状态流程

1. 管理员登录后台并选择目标用户。
2. 系统校验调用者必须为 `ADMIN`。
3. 系统更新目标用户的 `status`。
4. 若用户被设置为 `FROZEN` 或 `DISABLED`，则删除其现有 Redis 会话。
5. 系统记录 `operation_log` 审计日志。

## 6. 接口设计

所有接口均遵循统一响应结构：

```json
{
  "code": 0,
  "message": "success",
  "data": {}
}
```

### 6.1 `POST /api/auth/register`

用途：

- 注册普通用户账号

请求字段：

| 字段 | 类型 | 必填 | 说明 |
|---|---|---|---|
| username | string | 是 | 用户名，长度建议 4-50 |
| password | string | 是 | 明文密码，服务端负责哈希 |
| phone | string | 否 | 手机号 |
| email | string | 否 | 邮箱 |
| nickname | string | 否 | 昵称 |

响应字段：

| 字段 | 类型 | 说明 |
|---|---|---|
| userId | number | 新注册用户 ID |
| username | string | 用户名 |
| roleCode | string | 默认返回 `USER` |
| status | string | 默认返回 `ACTIVE` |

错误场景：

- 用户名、手机号或邮箱重复
- 密码格式不符合要求
- 请求字段缺失或非法

### 6.2 `POST /api/auth/login`

用途：

- 用户登录并获取 Token

请求字段：

| 字段 | 类型 | 必填 | 说明 |
|---|---|---|---|
| username | string | 是 | 用户名或登录标识 |
| password | string | 是 | 登录密码 |

响应字段：

| 字段 | 类型 | 说明 |
|---|---|---|
| token | string | 访问 Token |
| expireAt | string | 过期时间 |
| userInfo.userId | number | 用户 ID |
| userInfo.username | string | 用户名 |
| userInfo.roleCode | string | 角色 |
| userInfo.status | string | 账号状态 |

错误场景：

- 用户不存在
- 密码错误
- 账号冻结
- 账号禁用

### 6.3 `POST /api/auth/logout`

用途：

- 主动注销当前登录态

鉴权要求：

- 需要登录

请求字段：

- 无业务体，依赖请求头中的 Token

响应字段：

| 字段 | 类型 | 说明 |
|---|---|---|
| success | boolean | 是否登出成功 |

错误场景：

- Token 缺失
- Token 无效或已失效

### 6.4 `GET /api/auth/me`

用途：

- 获取当前登录用户上下文，用于前端页面初始化

鉴权要求：

- 需要登录

响应字段：

| 字段 | 类型 | 说明 |
|---|---|---|
| userId | number | 用户 ID |
| username | string | 用户名 |
| nickname | string | 昵称 |
| roleCode | string | 角色 |
| status | string | 账号状态 |

### 6.5 `PATCH /api/admin/users/{id}/status`

用途：

- 管理员修改目标用户账号状态

鉴权要求：

- 仅 `ADMIN`

请求字段：

| 字段 | 类型 | 必填 | 说明 |
|---|---|---|---|
| status | string | 是 | 目标状态，支持 `ACTIVE`、`FROZEN`、`DISABLED` |
| reason | string | 否 | 操作原因 |

响应字段：

| 字段 | 类型 | 说明 |
|---|---|---|
| userId | number | 目标用户 ID |
| oldStatus | string | 原状态 |
| newStatus | string | 新状态 |
| updatedAt | string | 更新时间 |

错误场景：

- 调用方无管理员权限
- 目标用户不存在
- 状态迁移非法

### 6.6 `GET /api/admin/users`

用途：

- 后台按角色或状态筛选用户

鉴权要求：

- `ADMIN`、`SUPPORT`

查询参数：

| 字段 | 类型 | 必填 | 说明 |
|---|---|---|---|
| roleCode | string | 否 | 按角色筛选 |
| status | string | 否 | 按状态筛选 |
| keyword | string | 否 | 按用户名、手机号模糊搜索 |
| pageNo | number | 否 | 页码 |
| pageSize | number | 否 | 每页条数 |

响应字段：

| 字段 | 类型 | 说明 |
|---|---|---|
| list | array | 用户列表 |
| total | number | 总数 |
| pageNo | number | 当前页码 |
| pageSize | number | 每页条数 |

## 7. 错误码设计

认证与权限模块建议使用 `4100-4199` 段：

| 错误码 | 说明 |
|---|---|
| 4101 | 用户名、手机号或邮箱已存在 |
| 4102 | 用户名或密码错误 |
| 4103 | Token 缺失或格式非法 |
| 4104 | Token 已过期 |
| 4105 | Token 无效或会话已失效 |
| 4106 | 账号已冻结 |
| 4107 | 账号已禁用 |
| 4108 | 权限不足 |
| 4109 | 状态迁移非法 |

同时仍遵循全局错误码原则：

- 参数错误可复用 `4001`
- 资源不存在可复用 `4002`
- 系统内部异常可复用 `5001`

## 8. 内部服务与依赖边界

### 8.1 内部服务接口

认证与权限模块建议对外提供 `IAuthService`：

| 方法 | 说明 |
|---|---|
| `RegisterUser` | 创建普通用户账号 |
| `Login` | 用户登录，返回 Token 和用户上下文 |
| `Logout` | 注销当前 Token |
| `GetCurrentUser` | 获取当前用户上下文 |
| `ValidateToken` | 校验 Token 并解析用户标识 |
| `RequireRole` | 校验角色是否满足访问要求 |
| `ChangeUserStatus` | 修改用户状态并清理会话 |

### 8.2 依赖边界

| 依赖对象 | 作用 | 边界说明 |
|---|---|---|
| `user_account` | 用户身份、角色、状态权威数据 | 认证模块只能修改自身相关字段，不处理订单或拍卖状态 |
| Redis 登录态缓存 | 存储 Token 对应会话 | 只作会话和加速用途，不替代 MySQL 用户权威数据 |
| `operation_log` | 记录管理员修改用户状态等敏感操作 | 必须可追踪操作人、目标用户、结果和时间 |
| 接入层鉴权过滤器 | 统一拦截请求并注入用户上下文 | 不承载业务规则，只做身份与权限前置校验 |

## 9. 关键状态流转

### 9.1 账号状态流转

```text
ACTIVE <-> FROZEN
ACTIVE -> DISABLED
FROZEN -> DISABLED
```

### 9.2 Token 状态流转

```text
签发 -> 有效 -> 登出失效
签发 -> 有效 -> 超时过期
签发 -> 有效 -> 管理员冻结/禁用后强制失效
```

## 10. 模块验收标准

`S04` 完成后，应满足以下标准：

- 用户可完成注册、登录、登出和获取当前用户信息
- 所有需要登录的接口具备统一 Token 鉴权入口
- 管理员、普通用户、运维/客服的权限边界清晰
- 账号冻结和禁用能阻断后续敏感操作
- 管理员修改用户状态会清理会话并留下审计日志
- 后续模块可以直接复用本模块提供的用户上下文和权限判断能力

## 11. 测试关注点

后续实现阶段，认证与权限模块至少应覆盖以下测试点：

- 注册成功、重复用户名、重复手机号
- 登录成功、密码错误、账号冻结、账号禁用
- Token 缺失、Token 过期、Token 已失效
- 普通用户访问管理员接口被拒绝
- `SUPPORT` 访问仅管理员接口被拒绝
- 管理员冻结用户后，该用户已有会话失效
- 登出后原 Token 不再可用

当前仓库仍处于文档设计阶段，本步未执行自动化测试。
