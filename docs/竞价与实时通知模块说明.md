# 在线拍卖平台竞价与实时通知模块说明

## 1. 文档目的

本文档用于完成 `S07`，明确竞价与实时通知模块在课程设计阶段的职责范围、业务流程、状态流转、接口契约、关键错误码、事务边界、锁策略和依赖关系。

本模块位于拍卖管理模块之后、订单与支付模块之前，负责在拍卖活动进入 `RUNNING` 后处理有效出价、维护当前最高价与最高出价者、执行延时保护，并输出竞价相关的实时通知能力。

## 2. 模块职责

竞价与实时通知模块负责以下能力：

- 买家对 `RUNNING` 活动发起出价
- 校验出价时间、金额格式、最小加价幅度、账号状态和请求幂等键
- 在同一事务内更新 `auction.current_price`、`auction.highest_bidder_id`、`auction.end_time` 和 `bid_record`
- 维护竞价历史和当前领先状态
- 在竞价成功后刷新热点缓存
- 发送价格更新、被超越提醒、延时保护广播等实时通知
- 为前台页面提供出价历史查询和个人当前竞价状态查询

本模块不负责以下内容：

- 不负责创建、取消和启动拍卖活动
- 不负责拍卖结束后的订单生成和支付处理
- 不负责通用消息中心、支付提醒等非竞价触发通知
- 不负责改变拍品和订单的终态结果

## 3. 核心实体与依赖

### 3.1 核心实体

本模块核心实体如下：

- `Auction`
- `BidRecord`
- `Notification`

### 3.2 主要依赖

| 依赖对象 | 作用 | 边界说明 |
|---|---|---|
| 认证与权限模块 | 识别当前用户身份、角色和账号状态 | 所有写接口必须先完成登录校验和状态校验 |
| `auction` 表 | 提供当前价、最高出价者、结束时间和活动状态 | 是最高价事实数据的权威来源 |
| `bid_record` 表 | 持久化有效出价历史 | 非法出价不写入 `bid_record` |
| `notification` 表 | 持久化站内通知记录 | 通知落库与推送不回滚竞价主事务 |
| Redis 缓存 | 提供热点活动价格和倒计时快照 | 当前代码先落地为可替换缓存抽象，默认实现为 `InMemoryBidCacheStore`，后续可无缝切到 Redis |
| WebSocket Gateway | 向订阅中的用户广播价格和状态变化 | 当前代码先落地为可替换推送抽象，默认实现为 `InMemoryAuctionEventGateway`，后续可替换为 Drogon WebSocket Gateway |
| 拍卖管理模块 | 提供活动状态和结束时间基线 | 本模块只接受 `RUNNING` 活动的有效出价 |
| 订单模块 | 消费拍卖结束后的最高出价结果 | 本模块不直接生成订单 |

### 3.3 Redis Key 约定

本模块直接依赖以下热点缓存键：

- `auction:detail:{auctionId}`
- `auction:price:{auctionId}`
- `auction:countdown:{auctionId}`

说明如下：

- `auction:price:{auctionId}` 只保存当前价、最高出价者摘要等热点读模型
- `auction:countdown:{auctionId}` 只保存剩余时间或延时保护辅助信息
- Redis 失效或不可用时必须回退 MySQL 查询，不能影响正确性

## 4. 角色与权限边界

### 4.1 普通用户权限

`USER` 在本模块中具备以下能力：

- 对处于 `RUNNING` 的活动提交出价
- 查看活动出价历史
- 查看自己在指定活动中的当前竞价状态
- 订阅活动价格和状态的实时更新

`USER` 不允许：

- 对未开始、已结束、已取消、已结算活动提交有效出价
- 手工修改活动当前价、最高出价者或结束时间
- 越过接口直接写入竞价记录

### 4.2 管理员权限

`ADMIN` 在本模块中具备以下能力：

- 只读查看出价历史和活动当前竞价状态
- 查看并发冲突、通知失败和重试记录
- 结合运维能力排查异常活动

`ADMIN` 不负责：

- 替用户代为出价
- 人工篡改合法最高价事实

### 4.3 客服/运维权限

`SUPPORT` 默认具备只读查询能力：

- 查看出价记录
- 查看通知投递结果
- 协助排查并发冲突和用户投诉

`SUPPORT` 不允许：

- 提交出价
- 修改最高价或竞价记录

## 5. 状态模型

### 5.1 出价记录状态定义

`bid_record.bid_status` 建议采用以下状态：

- `WINNING`：当前仍是活动的有效最高出价
- `OUTBID`：已被后续更高有效出价超越

### 5.2 有效出价判定规则

竞价模块只在以下条件同时满足时接受有效出价：

- `auction.status = RUNNING`
- 提交时刻必须早于活动当前 `end_time`
- 出价用户账号状态必须为 `ACTIVE`
- 出价金额必须为合法金额格式，且大于 0
- 若当前 `highest_bidder_id` 为空，则首个有效出价必须 `>= current_price`
- 若当前 `highest_bidder_id` 不为空，则新出价必须 `>= current_price + bid_step`

说明如下：

- `auction.current_price` 在活动创建时已初始化为 `start_price`
- 首次出价允许等于起拍价，后续出价必须至少满足最小加价幅度
- 即使活动状态尚未被调度任务切换，若当前时间已达到或超过 `end_time`，也必须拒绝新出价

### 5.3 实时通知事件定义

本步建议固定以下实时事件类型：

- `PRICE_UPDATED`：价格已刷新，广播给活动订阅者
- `OUTBID`：原最高出价者已被超越，定向提醒给对应用户
- `AUCTION_EXTENDED`：活动因延时保护被顺延，广播新的结束时间
- `AUCTION_CLOSED`：活动已结束，广播活动不可继续出价

说明：

- 活动开始和结束的调度触发方属于拍卖管理模块，但 WebSocket 事件结构可复用本模块约定
- 站内通知可复用 `notification.push_status` 的 `PENDING`、`SENT`、`FAILED` 状态定义

## 6. 业务流程

### 6.1 提交出价流程

1. 买家进入拍卖详情页并输入出价金额。
2. 前端为本次请求生成 `requestId` 并调用出价接口。
3. 系统先校验登录态、账号状态、请求参数和频率限制。
4. 系统可先读取 Redis 热点快照做预检查，但不能直接据此判定成功。
5. 系统开启数据库事务并锁定目标 `auction` 记录。
6. 系统在事务内再次校验活动状态、当前时间、最小可出价金额和请求幂等。
7. 系统更新上一条 `WINNING` 出价为 `OUTBID`，插入本次 `WINNING` 出价记录，并同步更新活动当前价、最高出价者和必要时的 `end_time`。
8. 事务提交成功后，系统刷新缓存并发出实时通知。

### 6.2 并发出价处理流程

1. 同一活动的多个出价请求同时到达时，统一竞争同一条 `auction` 行锁。
2. 获得锁的请求先完成事务校验和写入。
3. 后续请求在锁释放后必须读取最新 `current_price` 和 `end_time` 重新校验。
4. 若最新价格已变化导致金额不足，则返回并发冲突或最小可出价金额提示。
5. 并发失败请求不写入 `bid_record`。

### 6.3 延时保护流程

1. 事务内判断当前时间是否落入 `anti_sniping_window_seconds` 窗口。
2. 若命中窗口且当前活动配置开启延时保护，则在同一事务中把 `auction.end_time` 顺延 `extend_seconds`。
3. 顺延成功后，返回新的结束时间并广播 `AUCTION_EXTENDED` 事件。
4. 调度任务在结束活动前必须读取最新结束时间，避免误关拍卖。

### 6.4 出价历史查询流程

1. 前台或管理端请求活动出价历史。
2. 系统优先从 MySQL 查询按时间倒序的有效出价记录。
3. 返回出价金额、出价时间、状态和脱敏后的用户信息。
4. 历史查询不依赖 Redis，以保证可追溯性。

### 6.5 实时通知与降级流程

1. 竞价事务提交成功后，系统写入通知记录并向 WebSocket 订阅者推送事件。
2. 价格更新广播面向当前活动订阅者。
3. 被超越提醒优先定向发送给上一位最高出价者。
4. 若 WebSocket 不可用，则保留站内信，并由前端回退到轮询 `GET /api/auctions/{id}` 获取最新价格。
5. 通知失败进入重试或补偿路径，但不影响竞价事实提交。

## 7. 接口设计

所有接口遵循统一响应结构：

```json
{
  "code": 0,
  "message": "success",
  "data": {}
}
```

### 7.1 `POST /api/auctions/{id}/bids`

用途：

- 买家提交出价

鉴权要求：

- 需要登录
- 仅 `USER`

请求字段：

| 字段 | 类型 | 必填 | 说明 |
|---|---|---|---|
| requestId | string | 是 | 客户端幂等请求号 |
| bidAmount | number | 是 | 本次出价金额 |

响应字段：

| 字段 | 类型 | 说明 |
|---|---|---|
| bidId | number | 新出价记录 ID |
| auctionId | number | 活动 ID |
| bidAmount | number | 本次出价金额 |
| bidStatus | string | 默认 `WINNING` |
| currentPrice | number | 最新当前价 |
| currentHighestPrice | number | 最新当前最高价，当前与 `currentPrice` 保持一致 |
| highestBidderMasked | string | 脱敏后的当前领先用户 |
| endTime | string | 最新结束时间 |
| extended | boolean | 本次是否触发延时保护 |
| serverTime | string | 服务端受理时间 |

错误场景：

- 活动不存在
- 活动未开始或已结束
- 当前活动状态不允许出价
- 出价金额格式非法
- 出价金额未达到最小可出价金额
- 当前用户状态不允许出价
- 请求过于频繁
- 并发冲突导致当前价格已变化
- `requestId` 已存在且请求参数不一致

### 7.2 `GET /api/auctions/{id}/bids`

用途：

- 查询活动出价历史

访问规则：

- 公共活动可按活动可见性规则查询
- 管理员和客服可只读查询

查询参数：

| 字段 | 类型 | 必填 | 说明 |
|---|---|---|---|
| pageNo | number | 否 | 页码 |
| pageSize | number | 否 | 每页条数 |

响应字段建议包含：

- 出价记录列表
- `bidAmount`
- `bidTime`
- `bidStatus`
- `bidderMasked`
- 分页信息

### 7.3 `GET /api/auctions/{id}/my-bid`

用途：

- 买家查看自己在指定活动中的当前竞价状态

鉴权要求：

- 需要登录
- 仅 `USER`

响应字段建议包含：

- `auctionId`
- `userId`
- `myHighestBid`
- `isCurrentHighest`
- `lastBidTime`
- `currentPrice`
- `endTime`

### 7.4 `WebSocket /ws/auction/{id}`

用途：

- 订阅活动价格与状态变更的实时推送

接入要求：

- 建议登录后建立连接
- 未登录用户可通过轮询活动详情接口获取公开数据

事件消息体建议如下：

```json
{
  "event": "PRICE_UPDATED",
  "auctionId": 10001,
  "currentPrice": 128.00,
  "highestBidderMasked": "u***7",
  "endTime": "2026-04-12T20:30:15.123+08:00",
  "serverTime": "2026-04-12T20:25:11.456+08:00"
}
```

可选事件类型：

- `PRICE_UPDATED`
- `OUTBID`
- `AUCTION_EXTENDED`
- `AUCTION_CLOSED`

当前 `S22` 的浏览器接入使用轮询降级入口完成价格和被超越提示，后续如需替换为真实 Drogon WebSocket Gateway，可复用 `AuctionEventGateway` 的事件结构。

### 7.5 `GET /api/notifications`

用途：

- 查询当前登录用户的站内通知，作为 WebSocket 不可用时的轮询通知入口

鉴权要求：

- 需要登录

查询参数：

| 字段 | 类型 | 必填 | 说明 |
|---|---|---|---|
| limit | number | 否 | 最多返回条数，默认 20，最大 100 |
| unreadOnly | boolean | 否 | 是否只返回未读通知 |
| bizType | string | 否 | 业务类型，例如 `AUCTION` |
| bizId | number | 否 | 业务 ID，例如拍卖 ID |

说明：

- `S22` 买家拍卖详情使用 `bizType=AUCTION&bizId={auctionId}` 轮询被超越通知。
- `S24` 通知中心复用同一接口，可按全部、`AUCTION` 或 `ORDER` 过滤通知；订单评价通知使用 `bizType=ORDER&bizId={orderId}` 查询。
- 服务端始终按当前登录用户 ID 查询通知，不允许跨用户读取。

响应字段：

| 字段 | 类型 | 说明 |
|---|---|---|
| list | array | 通知列表 |
| total | number | 本次返回数量 |
| unreadCount | number | 本次返回结果中的未读数量 |
| limit | number | 本次查询上限 |

通知记录包含 `notificationId`、`noticeType`、`title`、`content`、`bizType`、`bizId`、`readStatus`、`pushStatus`、`createdAt` 和 `readAt`。

### 7.6 `PATCH /api/notifications/{id}/read`

用途：

- 将当前登录用户的一条站内通知标记为已读

鉴权要求：

- 需要登录

成功返回 `notificationId`、`readStatus` 和 `readAt`。

## 8. 事务边界与一致性要求

### 8.1 锁策略

课程设计阶段建议采用 MySQL 行级锁作为主锁策略：

- 对目标 `auction` 记录执行 `SELECT ... FOR UPDATE`
- 以 `auction` 为竞价聚合根串行化同一活动的并发写入
- 不使用 Redis 锁作为竞价正确性的前提

说明：

- `auction.version` 字段可保留给后续优化或乐观并发控制实验
- 当前阶段优先使用更直观、可验证的行级锁方案

### 8.2 出价主事务

出价成功必须在同一事务内完成以下动作：

- 锁定 `auction` 记录
- 再次校验 `auction.status = RUNNING`
- 再次校验当前时间早于最新 `end_time`
- 再次校验出价金额满足最小可出价条件
- 将上一条 `WINNING` 出价更新为 `OUTBID`
- 插入新的 `bid_record`，状态为 `WINNING`
- 更新 `auction.current_price`
- 更新 `auction.highest_bidder_id`
- 命中延时保护时更新 `auction.end_time`

上述动作必须同时成功或同时失败。

### 8.3 幂等与重复提交

幂等规则如下：

- 使用 `bid_record(auction_id, bidder_id, request_id)` 唯一约束控制重复提交
- 若同一 `requestId` 的请求已成功写入，则后续重复请求应返回同一业务结果，而不是再插入一条新出价
- 若同一 `requestId` 已存在但请求参数不一致，则返回幂等冲突错误

说明：

- 非法出价不写入 `bid_record`
- 幂等控制用于防止网络重试或前端重复点击导致的重复有效出价

### 8.4 缓存与通知边界

缓存与通知必须遵循以下顺序：

1. 先提交数据库事务
2. 再刷新或删除 `auction:detail:{auctionId}`、`auction:price:{auctionId}`、`auction:countdown:{auctionId}`
3. 再写入通知记录并向 WebSocket 推送

一致性要求：

- 不允许先更新缓存、后提交事务
- 通知失败不回滚竞价事务，只记录失败并进入重试
- Redis 不可用时，前台读取回退 MySQL，出价主事务仍以数据库为准

## 9. 错误码设计

竞价与实时通知模块建议使用 `4400-4499` 段：

| 错误码 | 说明 |
|---|---|
| 4401 | 拍卖活动不存在 |
| 4402 | 当前活动未开始 |
| 4403 | 当前活动已结束或已进入结算 |
| 4404 | 当前活动状态不允许出价 |
| 4405 | 出价金额格式非法 |
| 4406 | 出价金额未达到最小可出价金额 |
| 4407 | 当前用户状态不允许出价 |
| 4408 | 请求幂等键冲突 |
| 4409 | 请求过于频繁，请稍后重试 |
| 4410 | 并发出价冲突，请刷新最新价格后重试 |
| 4411 | 实时订阅通道无效 |
| 4412 | 出价记录查询参数非法 |

同时可复用全局错误码：

- `4001`：参数非法
- `4002`：资源不存在
- `4004`：权限不足
- `5001`：内部服务异常

## 10. 内部服务设计

当前代码已落地以下服务与抽象：

| 类型/方法 | 说明 |
|---|---|
| `BidService::PlaceBid` | 提交出价并维护最高价、最高出价者、延时保护和幂等返回 |
| `BidService::ListAuctionBidHistory` | 查询活动出价历史 |
| `BidService::GetMyBidStatus` | 查询当前用户在指定活动中的竞价状态 |
| `NotificationService::PublishBidEvents` | 在事务提交后发送价格更新、被超越提醒和延时保护广播 |
| `NotificationService::ListUserNotifications` | 查询当前用户站内通知，为浏览器轮询提供降级入口 |
| `NotificationService::MarkUserNotificationRead` | 将当前用户通知标记为已读 |
| `BidCacheStore` / `InMemoryBidCacheStore` | 承载热点价格快照缓存，保证缓存失败不影响主事务 |
| `AuctionEventGateway` / `InMemoryAuctionEventGateway` | 承载活动广播和用户定向推送，保证推送失败不回滚出价 |

说明：

- 当前仓库尚未接入 Redis C++ 客户端和 Drogon WebSocket 运行环境，因此先以可替换抽象完成模块边界和自动化验证
- `BidService` 负责提供竞价事实和事件上下文，通知模块负责落库和发送

## 11. 模块验收标准

`S07` 完成后，应满足以下标准：

- 同一活动在任一时刻只有一个合法当前最高价和一个最高出价者
- 出价记录写入、最高价更新、最高出价者更新、延时保护更新时间在同一事务内完成
- 非法出价和并发失败出价不会污染 `bid_record`
- 活动结束后即使状态尚未切换，也不会再接受新的有效出价
- 延时保护触发后，后续读取和调度都能看到最新 `end_time`
- 价格更新、被超越提醒和 WebSocket 广播不会阻塞主事务提交

## 12. 测试关注点

本次代码实现已实际覆盖以下测试点：

- 首次出价等于起拍价时成功
- 后续出价未达到 `current_price + bid_step` 时失败
- 同一活动最小并发出价时最终只有一个成功写入
- 重复 `requestId` 重试不会产生重复有效出价
- 触发延时保护时 `end_time` 在事务中被正确顺延
- 被超越的上一条 `WINNING` 记录会在同一事务中改为 `OUTBID`
- WebSocket 或通知失败不影响竞价事务提交

当前尚未单独补专项自动化覆盖，但模块设计和实现已预留相应边界：

- 非 `RUNNING` 状态出价被拒绝
- `end_time` 边界上的出价被正确拒绝
- 相同 `requestId` 但参数不一致时返回幂等冲突
- Redis 不可用时仍可基于 MySQL 完成正确竞价
