# 在线拍卖平台订单与支付模块说明

## 1. 文档目的

本文档用于完成 `S08`，明确订单与支付结算模块在课程设计阶段的职责范围、状态机、事务边界、接口契约、幂等处理、异常分支和验证方式。

本模块消费 `S07` 已产出的最高价与 `SETTLING` 活动结果，负责把拍卖结果推进到“待支付”“已支付”或“已关闭”闭环，但不负责评价、统计和运维重试。

## 2. 模块职责

本模块负责以下能力：

- 扫描 `auction.status = SETTLING` 的活动并生成唯一订单
- 通过 `order_info.auction_id` 唯一约束保证一场拍卖最多生成一笔订单
- 买家发起支付并生成支付流水
- 校验支付回调签名、金额、商户号、订单号和流水号
- 对重复成功回调、失败回调和乱序回调做幂等处理
- 对超时未支付订单执行关闭，并把活动与拍品推进到 `UNSOLD`
- 为结算建单、支付回调和超时关闭写入 `operation_log`、`payment_callback_log` 和 `task_log`
- 在事务提交后补发站内通知

本模块不负责以下内容：

- 不负责拍卖活动开始、结束调度
- 不负责修改历史出价记录
- 不负责卖家发货、买家确认收货和评价流转
- 不负责真实第三方支付网关接入，本步仅落地可验证的 mock 支付契约

## 3. 核心实体与依赖

### 3.1 核心实体

- `Auction`
- `OrderInfo`
- `PaymentRecord`
- `PaymentCallbackLog`
- `Notification`

### 3.2 主要依赖

| 依赖对象 | 作用 | 边界说明 |
|---|---|---|
| 拍卖管理模块 | 提供 `SETTLING` 活动、最高出价者和最终成交价 | 订单模块只消费结束态结果，不反写出价记录 |
| 认证与权限模块 | 校验支付发起用户身份和账号状态 | 只有订单买家可以发起支付 |
| `order_info` 表 | 保存订单事实和履约状态 | `auction_id` 全局唯一 |
| `payment_record` 表 | 保存支付发起和回调处理结果 | 同一订单允许多次支付尝试，但同一支付单只处理一次成功 |
| `payment_callback_log` 表 | 保存每次回调的接收、校验和处理结果 | 支持排障和人工补单 |
| `notification` 表 | 保存待支付、支付成功、超时关闭等站内通知 | 通知失败不回滚订单或支付事实 |
| `task_log` 表 | 保存建单和超时关闭批处理结果 | 用于补偿和演示 |
| `operation_log` 表 | 保存结算、支付发起和支付回调审计日志 | 便于答辩时展示关键操作轨迹 |

## 4. 角色与权限边界

### 4.1 普通用户

`USER` 在本模块中具备以下能力：

- 获胜买家对属于自己的 `PENDING_PAYMENT` 订单发起支付
- 接收待支付、支付成功、支付失败和订单关闭通知

`USER` 不允许：

- 对不属于自己的订单发起支付
- 对已关闭、已支付或超过支付截止时间的订单再次发起支付
- 直接回写订单、支付、活动终态

### 4.2 管理员 / 客服

`ADMIN` 与 `SUPPORT` 当前只承担只读和排障职责：

- 查询订单、支付、回调日志和任务日志
- 核对支付状态与异常订单

本步未开放管理员代付、强制改价、手工改回调等高风险写操作。

## 5. 状态模型

### 5.1 拍卖与拍品联动

本步继续承接 `S06-S07` 的状态约束：

- `RUNNING -> SETTLING`：活动结束且存在最高出价
- `SETTLING -> SOLD`：支付成功，成交成立
- `SETTLING -> UNSOLD`：超时未支付或结算失败

拍品状态协同如下：

- 活动进入 `SETTLING` 时，拍品保持 `IN_AUCTION`
- 支付成功时，拍品更新为 `SOLD`
- 超时关闭时，拍品更新为 `UNSOLD`

### 5.2 订单状态

本步实际使用以下订单状态：

- `PENDING_PAYMENT`
- `PAID`
- `CLOSED`

后续履约与评价阶段保留：

- `WAITING_DELIVERY`
- `SHIPPED`
- `COMPLETED`
- `REVIEWED`

状态流转如下：

```text
[*] -> PENDING_PAYMENT
PENDING_PAYMENT -> PAID
PENDING_PAYMENT -> CLOSED
```

### 5.3 支付状态

本步实际使用以下支付状态：

- `WAITING_CALLBACK`
- `SUCCESS`
- `FAILED`
- `CLOSED`

保留扩展状态：

- `INITIATED`

状态流转如下：

```text
[*] -> WAITING_CALLBACK
WAITING_CALLBACK -> SUCCESS
WAITING_CALLBACK -> FAILED
WAITING_CALLBACK -> CLOSED
FAILED -> CLOSED
```

### 5.4 回调日志状态

`payment_callback_log.callback_status` 使用如下状态：

- `RECEIVED`
- `VERIFIED`
- `REJECTED`
- `PROCESSED`

## 6. 业务流程

### 6.1 结算建单流程

1. 调度扫描全部 `SETTLING` 活动。
2. 对每个活动开启事务并锁定 `auction` 与关联 `item`。
3. 再次确认活动仍处于 `SETTLING`，且尚未存在 `order_info` 记录。
4. 若缺失 `highest_bidder_id`，回退为 `UNSOLD` 并记录审计日志。
5. 若存在最高出价者，则创建 `PENDING_PAYMENT` 订单，并写入支付截止时间。
6. 事务提交后向买家和卖家写入待支付站内通知。

### 6.2 支付发起流程

1. 买家携带 Bearer Token 调用支付发起接口。
2. 系统校验订单归属、订单状态和支付截止时间。
3. 若订单已有同渠道活跃支付单，则直接返回已有支付单，避免重复创建。
4. 若无活跃支付单，则创建新的 `payment_record`，状态为 `WAITING_CALLBACK`。
5. 返回 mock 支付地址、支付单号和商户号。

### 6.3 支付回调流程

1. 支付模块根据 `payment_no` 锁定支付单、订单、活动和拍品。
2. 先写入 `payment_callback_log(RECEIVED)`。
3. 校验 `order_no`、`merchant_no`、金额、流水号冲突和签名。
4. 对校验失败的回调更新为 `REJECTED`，仅保留日志，不回写业务主状态。
5. 对首次成功回调，更新 `payment_record = SUCCESS`、`order_info = PAID`、`auction = SOLD`、`item = SOLD`。
6. 对重复成功回调，只更新日志为 `PROCESSED`，不重复推进状态。
7. 对失败回调，更新支付单为 `FAILED`，订单保持 `PENDING_PAYMENT`，允许在截止时间前重试支付。

### 6.4 超时关闭流程

1. 调度扫描全部已过 `pay_deadline_at` 且仍为 `PENDING_PAYMENT` 的订单。
2. 锁定订单、活动和拍品，并再次校验截止时间确已到达。
3. 关闭该订单下仍未终态的支付单。
4. 把订单更新为 `CLOSED`，并把活动和拍品推进到 `UNSOLD`。
5. 事务提交后通知买家和卖家订单已关闭。

## 7. 接口设计

所有接口遵循统一响应结构：

```json
{
  "code": 0,
  "message": "success",
  "data": {}
}
```

### 7.1 `POST /api/orders/{id}/pay`

用途：

- 获胜买家发起支付

鉴权要求：

- 需要登录
- 仅 `USER`
- 必须是订单 `buyer_id`

请求字段：

| 字段 | 类型 | 必填 | 说明 |
|---|---|---|---|
| payChannel | string | 是 | `MOCK_ALIPAY` 或 `MOCK_WECHAT` |

成功响应示例：

```json
{
  "code": 0,
  "message": "success",
  "data": {
    "orderId": 12001,
    "orderNo": "ORD1744710000000000",
    "paymentId": 9001,
    "paymentNo": "PAY1744710000000100",
    "payChannel": "MOCK_ALIPAY",
    "payAmount": 150.0,
    "payStatus": "WAITING_CALLBACK",
    "merchantNo": "COURSE_AUCTION",
    "payUrl": "mockpay://checkout?...",
    "expireAt": "2026-04-15 14:30:00",
    "reusedExisting": false
  }
}
```

### 7.2 `POST /api/payments/callback`

用途：

- 接收第三方支付回调

当前 mock 回调字段：

| 字段 | 类型 | 必填 | 说明 |
|---|---|---|---|
| paymentNo | string | 是 | 平台支付单号 |
| orderNo | string | 是 | 订单号 |
| merchantNo | string | 是 | 当前固定为 `COURSE_AUCTION` |
| transactionNo | string | 是 | 第三方流水号 |
| payStatus | string | 是 | `SUCCESS` 或 `FAILED` |
| payAmount | number | 是 | 回调金额 |
| signature | string | 是 | HMAC-SHA256 签名 |

签名规则：

```text
HMAC_SHA256(secret, merchantNo|orderNo|paymentNo|transactionNo|payStatus|payAmount)
```

当前实现说明：

- `secret` 复用当前配置中的 `auth.token_secret`
- 这是课程设计阶段的 mock 支付契约，后续接真实支付网关时替换签名与网关字段即可

### 7.3 系统批处理入口

当前环境未接入真实 HTTP 调度入口，批处理通过 service + scheduler 直接调用：

- `OrderScheduler::RunSettlementCycle()`
- `OrderScheduler::RunTimeoutCloseCycle()`

## 8. 事务边界与幂等规则

### 8.1 结算建单事务

以下动作必须在同一事务内完成：

- 锁定 `auction` / `item`
- 校验 `SETTLING` 状态
- 判断订单是否已存在
- 创建 `order_info`

事务提交后才允许发通知。

### 8.2 支付回调事务

以下动作必须在同一事务内完成：

- 锁定 `payment_record` / `order_info` / `auction` / `item`
- 插入回调日志
- 验签与金额校验
- 推进支付、订单、活动、拍品状态
- 写入审计日志

### 8.3 幂等控制

- `order_info.auction_id` 唯一约束保证一场拍卖最多生成一笔订单
- 同订单同渠道已有活跃支付单时，支付发起直接复用已有记录
- 支付成功回调再次到达时只写日志，不重复记账
- 已关闭订单收到失败回调时只记录日志，不重新打开订单
- `payment_record.transaction_no` 唯一约束防止同流水号重复入账

## 9. 关键错误码

| 错误码 | 含义 |
|---|---|
| `4501` | 订单不存在 |
| `4502` | 订单状态不允许当前操作 |
| `4503` | 当前用户不是订单买家 |
| `4504` | 支付截止时间已过 |
| `4505` | 订单结算状态冲突 |
| `4601` | 支付单不存在 |
| `4602` | 支付渠道非法 |
| `4603` | 支付状态冲突 |
| `4604` | 支付回调签名非法 |
| `4605` | 支付金额不匹配 |
| `4606` | 支付回调字段非法 |
