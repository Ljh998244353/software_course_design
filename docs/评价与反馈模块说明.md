# 在线拍卖平台评价与反馈模块说明

## 1. 文档目的

本文档用于完成 `S09`，明确评价与反馈模块在课程设计阶段的职责范围、订单状态约束、互评规则、信用汇总口径、接口契约和验证方式。

本模块位于订单与支付模块之后，负责消费已完成订单并输出买卖双方互评、订单评价查询和用户信用汇总能力。

## 2. 模块职责

本模块负责以下能力：

- 买家和卖家针对同一已完成订单互相提交一次评价
- 自动根据订单角色推导评价方向，不接受客户端伪造 `review_type`
- 基于 `(order_id, reviewer_id)` 唯一约束阻止重复评价
- 查询指定订单下的互评记录
- 汇总指定用户收到的评价数量、平均分和正中差评计数
- 在第二条评价提交成功后把订单状态从 `COMPLETED` 推进到 `REVIEWED`
- 为评价提交写入 `operation_log`，并在事务提交后补发站内通知

本模块不负责以下内容：

- 不负责订单支付、超时关闭或拍卖成交判定
- 不负责卖家发货和买家确认收货等履约流程
- 不负责统计报表、运维补偿和异常重试
- 不负责公开搜索或复杂分页筛选，本步只落地最小查询能力

## 3. 核心实体与依赖

### 3.1 核心实体

- `OrderInfo`
- `Review`
- `Notification`

### 3.2 主要依赖

| 依赖对象 | 作用 | 边界说明 |
|---|---|---|
| 认证与权限模块 | 校验当前登录用户身份和账号状态 | 只有 `USER` 可提交评价 |
| 订单模块 | 提供订单归属、状态和订单号 | `S12` 已补齐支付后发货/确认收货链路，本模块只消费 `COMPLETED/REVIEWED` 状态 |
| `review` 表 | 持久化评分、评论和评价方向 | `(order_id, reviewer_id)` 全局唯一 |
| `notification` 表 | 保存收到新评价提醒 | 通知失败不回滚评价事实 |
| `operation_log` 表 | 保存评价提交审计日志 | 用于排障和答辩展示 |

## 4. 角色与约束

### 4.1 普通用户

`USER` 在本模块中具备以下能力：

- 买家对订单卖家提交评价
- 卖家对订单买家提交评价
- 查询指定订单下的互评结果
- 查询指定用户的信用汇总

`USER` 不允许：

- 对非本人参与的订单提交评价
- 对 `PENDING_PAYMENT`、`PAID`、`CLOSED` 等未完成订单提交评价
- 对同一订单重复提交第二次评价
- 伪造评价方向或把自己作为被评价人

### 4.2 管理员 / 客服

本步未开放 `ADMIN`、`SUPPORT` 的评价写接口；后续若需要介入纠纷处理，应通过单独运维入口完成，不直接复用普通用户评价接口。

## 5. 状态模型

### 5.1 订单状态约束

本模块只接受以下订单状态：

- `COMPLETED`
- `REVIEWED`

状态流转如下：

```text
COMPLETED --首条评价--> COMPLETED
COMPLETED --第二条评价--> REVIEWED
REVIEWED --补查/已完成查询--> REVIEWED
```

说明：

- 第一条评价写入成功后，订单仍保持 `COMPLETED`
- 第二条评价写入成功后，订单更新为 `REVIEWED`
- `S12` 已通过订单服务补齐 `PAID -> SHIPPED -> COMPLETED` 的最小履约链路，评价模块不再依赖 SQL 夹具推进订单状态

### 5.2 评价方向定义

- `BUYER_TO_SELLER`
- `SELLER_TO_BUYER`

方向由服务端根据 `order_info.buyer_id` / `order_info.seller_id` 自动推导。

## 6. 业务流程

### 6.1 提交评价流程

1. 用户携带 Bearer Token 调用评价提交接口。
2. 系统校验评分范围、评论长度和登录身份。
3. 系统开启事务并锁定目标订单。
4. 再次确认订单存在、当前用户是买家或卖家，且订单状态为 `COMPLETED/REVIEWED`。
5. 系统查询 `(order_id, reviewer_id)` 是否已有评价，若存在则直接拒绝。
6. 系统根据订单归属自动推导 `review_type` 与 `reviewee_id`，插入评价记录。
7. 若该订单评价数达到 2，则把订单状态更新为 `REVIEWED`。
8. 事务提交后向被评价人补发收到新评价提醒。

### 6.2 订单评价查询流程

1. 调用方根据 `order_id` 查询订单。
2. 系统先校验订单是否存在。
3. 系统按创建时间顺序返回该订单下的全部评价记录。

### 6.3 用户信用汇总流程

1. 调用方根据 `user_id` 查询信用汇总。
2. 系统聚合该用户收到的评价总数、平均分、正评数、中评数和差评数。
3. 当前口径定义为 `4-5` 分正评、`3` 分中评、`1-2` 分差评。

## 7. 接口设计

所有接口遵循统一响应结构：

```json
{
  "code": 0,
  "message": "success",
  "data": {}
}
```

### 7.1 `POST /api/reviews`

用途：

- 买卖双方对已完成订单提交评价

鉴权要求：

- 需要登录
- 仅 `USER`
- 当前用户必须是订单买家或卖家

请求字段：

| 字段 | 类型 | 必填 | 说明 |
|---|---|---|---|
| orderId | number | 是 | 订单 ID |
| rating | number | 是 | 评分，范围 `1-5` |
| content | string | 否 | 评价内容，最大 500 字符 |

成功响应示例：

```json
{
  "code": 0,
  "message": "success",
  "data": {
    "reviewId": 9101,
    "orderId": 12001,
    "reviewType": "BUYER_TO_SELLER",
    "reviewerId": 30001,
    "revieweeId": 30002,
    "rating": 5,
    "content": "Smooth transaction and fast delivery.",
    "orderStatusAfter": "COMPLETED",
    "orderMarkedReviewed": false
  }
}
```

### 7.2 `GET /api/orders/{id}/reviews`

用途：

- 查询指定订单下的互评记录

返回字段：

- `orderId`
- `orderStatus`
- `reviews[]`

### 7.3 `GET /api/users/{id}/reviews/summary`

用途：

- 查询指定用户的信用汇总

返回字段：

- `userId`
- `totalReviews`
- `averageRating`
- `positiveReviews`
- `neutralReviews`
- `negativeReviews`

## 8. 关键错误码

| 错误码 | 含义 | 触发场景 |
|---|---|---|
| `4501` | `order not found` | 订单不存在 |
| `4701` | `review rating invalid` | 评分不在 `1-5` |
| `4702` | `review content too long` | 评论长度超过 500 |
| `4703` | `review order status invalid` | 订单未达到 `COMPLETED/REVIEWED` |
| `4704` | `review duplicate submitted` | 同一订单重复评价 |
| `4705` | `review permission denied` | 当前用户不是该订单买家或卖家 |
