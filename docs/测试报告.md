# 在线拍卖平台测试报告

## 1. 报告范围

本文档记录 `S14` 高风险专项测试的实际落地结果，覆盖竞价、拍卖结束、订单结算、支付回调、缓存降级、通知失败和安全负向路径。

本次测试仍基于当前仓库真实实现执行：`CTest + 断言式测试二进制 + Shell 脚本 + 本地 MySQL`。Drogon HTTP 控制器尚未批量接入，因此本报告不声称完成真实 HTTP 接口级自动化覆盖。

## 3. 场景结果

| 风险点 | 验证场景 | 结果 |
|---|---|---|
| 并发出价最高价一致性 | 12 个买家同时以相同金额出价，同一拍卖最终只允许 1 条 `WINNING` 出价和 1 条有效出价记录 | 通过 |
| 拍卖结束竞争 | 拍卖结束调度与晚到出价并发执行，晚到出价返回 `kBidAuctionClosed`，活动进入 `SETTLING` | 通过 |
| 订单结算竞争 | 4 个结算任务并发扫描同一 `SETTLING` 拍卖，依赖唯一约束保证一场拍卖只生成 1 笔订单 | 通过 |
| 支付回调幂等 | 6 个相同成功回调并发到达，仅 1 个非幂等生效，其余返回幂等结果，订单最终为 `PAID` | 通过 |
| 成功后失败回调 | 支付成功后再收到失败回调，系统保持订单 `PAID` 并幂等忽略 | 通过 |
| 缓存降级 | 出价成功后模拟竞价缓存刷新失败，数据库最高价和出价事实仍提交成功，后续出价可刷新缓存 | 通过 |
| 通知失败 | 模拟被超越用户直推失败，出价事实不回滚，失败通知持久化后可重试为 `SENT` | 通过 |
| 未认证写入 | 缺失 Bearer Token 的出价请求被拒绝 | 通过 |
| RBAC 越权 | 普通用户调用管理员创建拍卖能力被拒绝 | 通过 |
| 冻结账号 | 管理员冻结用户后旧会话失效，重新登录返回冻结错误 | 通过 |
| 出价输入校验 | 三位小数金额被拒绝，相同幂等键不同金额被拒绝，结束后出价被拒绝 | 通过 |
| 支付回调篡改 | 商户号、金额、签名、已处理流水号篡改均被拒绝并写入拒绝回调日志 | 通过 |

## 4. 缺陷闭环

本次专项测试发现并修复 1 个真实缺陷：

- 缺陷：`BidService` 的金额小数位校验把输入值和归一化值都再次归一化，导致 `100.001` 这类三位小数金额可能被接受。
- 修复：在 `src/modules/bid/bid_service.cpp` 中新增独立的小数位校验逻辑，直接校验 `value * 100` 是否接近整数。
- 回归：`auction_security_negative` 已覆盖三位小数出价拒绝场景。

环境说明：

- 在当前 Codex 沙箱内启动本地 MySQL Unix socket 会被限制，最初的 `scripts/test.sh risk` 未进入业务断言阶段。
- 按权限流程在沙箱外重新运行后，本地 MySQL 正常启动，专项测试全部通过。
