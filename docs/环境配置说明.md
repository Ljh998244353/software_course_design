# 在线拍卖平台环境配置说明

## 1. 文档目的

本文档用于完成 `S02`，明确本项目在课程设计阶段的开发环境、依赖组件、工程骨架、配置文件约定、日志与错误码基线，以及本地初始化步骤。

本文档的目标不是覆盖业务设计，而是保证项目在进入 `S03` 及后续模块开发前，具备统一、可复现、可交接的工程起点。

## 2. 适用范围

适用范围如下：

- 开发环境：WSL2 Ubuntu 24.04
- Shell 环境：`zsh`
- 项目形态：浏览器访问的 C++ 分层单体应用
- 后端基线：`C++20 + CMake + Drogon + MySQL + Redis + Nginx`

## 3. 环境基线

### 3.1 操作系统与基础工具

建议统一以下基线：

- 操作系统：WSL2 Ubuntu 24.04
- 编译器：GCC 或 Clang，要求支持 C++20
- 构建工具：CMake
- 版本管理：Git
- 常用工具：`make`、`pkg-config`、`curl`、`openssl`

建议要求：

- 编译器、构建工具和依赖管理方式在团队内保持一致
- 所有开发成员在开始编码前先完成本地编译链验证
- 本地环境异常时，优先修正环境问题，不绕过编译与依赖检查

### 3.2 运行时组件

课程设计阶段建议统一使用以下运行时组件：

- Web 框架：Drogon
- 数据库：MySQL 8
- 缓存：Redis 7
- 反向代理：Nginx
- 文件存储：本地磁盘目录，后续可替换为对象存储

组件职责约束如下：

- MySQL 是拍卖、出价、订单、支付等交易事实的唯一权威数据源
- Redis 只用于缓存、热点读取和短期状态，不负责最终一致性判定
- Nginx 负责静态资源分发、反向代理和统一入口
- 图片和附件优先采用本地文件存储，避免在课程设计阶段过早引入外部对象存储复杂度

## 4. 工程骨架

### 4.1 目录结构约定

项目开始编码时，建议采用以下目录结构：

```text
.
├── CMakeLists.txt
├── src
│   ├── access
│   │   ├── http
│   │   └── ws
│   ├── application
│   ├── domain
│   ├── repository
│   ├── infra
│   │   ├── cache
│   │   ├── payment
│   │   └── storage
│   ├── jobs
│   └── common
├── tests
│   ├── unit
│   ├── integration
│   ├── concurrency
│   └── e2e
├── assets
├── config
├── scripts
├── sql
└── docs
```

各目录职责如下：

- `src/access/http`：REST 控制器、请求校验、统一响应封装
- `src/access/ws`：WebSocket 会话和实时推送入口
- `src/application`：业务编排、事务边界、跨模块协作
- `src/domain`：领域规则、状态机、核心实体与校验
- `src/repository`：MySQL 数据访问与持久化实现
- `src/infra/cache`：Redis 访问与缓存失效
- `src/infra/payment`：支付适配器与回调验签
- `src/infra/storage`：本地文件或对象存储适配
- `src/jobs`：拍卖结束、补偿、统计、超时关闭任务
- `src/common`：配置、错误码、日志、时间、ID、工具函数
- `tests`：测试代码，按单元、集成、并发和端到端分层
- `assets`：浏览器端静态资源，当前包含 `assets/demo/` 前端演示台
- `config`：本地配置模板、应用配置、Nginx 配置
- `scripts`：环境初始化、启动、测试和清理脚本
- `sql`：建表脚本、索引脚本、初始化数据

### 4.2 代码组织要求

工程骨架需遵循以下原则：

- 控制器不直接承载复杂业务规则
- 事务边界放在应用层，而不是控制器层
- 领域规则不依赖 HTTP 细节
- MySQL 写路径必须保留清晰的数据边界
- 缓存更新必须发生在数据库事务成功之后

## 5. 配置文件约定

### 5.1 推荐配置目录

建议统一以下配置文件位置：

```text
config/
├── app.example.json
├── app.local.json
├── nginx.conf
└── logging.example.json
```

说明如下：

- `app.example.json`：提交到仓库，作为默认模板
- `app.local.json`：本地实际配置，不提交敏感内容
- `nginx.conf`：Nginx 本地或实验环境配置模板
- `logging.example.json`：日志级别、输出路径、格式字段模板

### 5.2 应用配置模板建议

建议应用配置至少覆盖以下字段：

```json
{
  "server": {
    "host": "0.0.0.0",
    "port": 8080
  },
  "mysql": {
    "host": "127.0.0.1",
    "port": 3306,
    "database": "auction_system",
    "username": "auction_user",
    "password": "change_me",
    "socket": "",
    "charset": "utf8mb4",
    "connect_timeout_seconds": 3
  },
  "redis": {
    "host": "127.0.0.1",
    "port": 6379,
    "db": 0
  },
  "storage": {
    "type": "local",
    "base_path": "./data/uploads"
  },
  "auth": {
    "token_expire_minutes": 120,
    "token_secret": "change_me_auth_secret",
    "password_hash_rounds": 100000
  },
  "logging": {
    "level": "info",
    "path": "./logs/app.log"
  }
}
```

配置约束如下：

- 密码、密钥和签名参数不得写死在代码中
- 本地配置与模板配置字段名必须保持一致
- 每新增一个外部依赖，都必须先在模板中声明
- 不允许让模块依赖隐式环境变量才能运行
- 当使用本地 Unix socket 连接 MySQL 时，应填写 `mysql.socket`，并保持 `mysql.charset` 与库表字符集一致

## 6. 日志与错误码基线

### 6.1 日志基线

课程设计阶段至少应支持以下日志类型：

- 应用日志：记录接口请求、业务处理结果、依赖调用结果
- 审计日志：记录管理员审核、权限变更、支付处理等敏感操作
- 任务日志：记录拍卖结束、超时关闭、统计任务、补偿任务执行结果

建议日志字段至少包含：

- `trace_id`
- `user_id`
- `role`
- `module`
- `event`
- `auction_id`
- `order_id`
- `result`
- `error_code`
- `latency_ms`
- `created_at`

### 6.2 错误码基线

错误码建议按以下规则统一：

- `0`：成功
- `4xxx`：参数错误、业务状态错误、权限错误、认证失败
- `5xxx`：系统内部错误、数据库异常、缓存异常、外部依赖异常

建议预留以下错误码段：

- `4100-4199`：认证与权限
- `4200-4299`：物品与审核
- `4300-4399`：拍卖管理
- `4400-4499`：竞价与通知
- `4500-4599`：订单与支付
- `4600-4699`：评价与统计
- `5000-5099`：系统与基础设施异常

## 7. 本地初始化步骤

### 7.1 初始化顺序

建议 agent 或开发成员按以下顺序执行本地初始化：

1. 安装编译工具链与基础依赖
2. 安装并启动 MySQL、Redis、Nginx
3. 初始化数据库库名、账号和基础权限
4. 准备 `config/app.local.json`
5. 创建 `logs/`、`data/uploads/`、`build/` 等本地目录
6. 执行 CMake 配置与构建
7. 启动应用并执行基础健康检查

### 7.2 建议命令接口

项目进入编码阶段后，建议统一以下命令接口：

```bash
cmake -S . -B build
cmake --build build
ctest --test-dir build
./build/bin/auction_app
./build/bin/auction_app --check-config
./build/bin/auction_app --check-db
```

数据库初始化与配置检查建议保留对应脚本接口，例如：

```bash
./scripts/bootstrap.sh
./scripts/db/setup_local_mysql.sh
./scripts/run_local.sh
./scripts/test.sh
./scripts/test_db.sh
./scripts/test_auth.sh
./scripts/deploy/init_demo_env.sh
./scripts/deploy/run_demo_server.sh
./scripts/deploy/verify_release.sh
./scripts/test.sh frontend
```

说明：

- 当前仓库已完成 `S02` 工程骨架落地，上述命令接口已可直接使用
- 后续步骤应继续复用这些统一脚本，而不是让每位成员手工执行零散命令
- `S15` 交付阶段统一使用 `scripts/deploy/` 下的脚本完成演示环境初始化、服务启动和最终回归验证
