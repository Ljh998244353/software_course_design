# 在线拍卖平台系统监控与异常处理模块说明

## 1. 文档目的

本文档用于说明 `S11` 已落地的系统监控与异常处理模块实现，包括日志查询、异常看板、通知失败重试、手工异常标记和补偿入口。

当前本模块以服务层能力交付，尚未单独接入 Drogon HTTP 控制器。

## 2. 模块职责

本模块负责以下能力：

- 查询 `operation_log` 操作日志
- 查询 `task_log` 任务日志
- 聚合 `notification`、`task_log`、`payment_callback_log` 和手工标记，形成最小异常看板
- 对失败的通知推送执行重试
- 记录 Redis 降级、人工排障等手工异常标记
- 提供订单结算、订单超时关闭、统计日报重跑和通知补偿入口

本模块不负责以下内容：

- 不负责新增交易事实表
- 不直接改写竞价、订单、支付主状态机规则
- 不负责完整告警平台、消息订阅或外部监控系统集成
- 不负责 Drogon 控制器与真实管理后台页面

## 3. 核心依赖与数据来源

### 3.1 复用表

- `operation_log`
- `task_log`
- `notification`
- `payment_callback_log`

### 3.2 复用服务

| 依赖对象 | 作用 |
|---|---|
| `AuthMiddleware` | 鉴权与角色校验 |
| `NotificationService` | 失败通知重试 |
| `OrderService` | 订单结算补偿、订单超时关闭补偿 |
| `StatisticsService` | 指定日期日报重跑 |

### 3.3 角色约束

- `ADMIN`：可查询日志、查看异常、标记异常、执行通知重试与补偿
- `SUPPORT`：可查询日志、查看异常、标记异常
- `USER`：无权访问本模块

## 4. 业务流程

### 4.1 操作日志查询

1. `ADMIN` 或 `SUPPORT` 发起查询
2. 系统校验 `module_name`、`result` 与 `limit`
3. 按时间倒序返回 `operation_log`

### 4.2 任务日志查询

1. `ADMIN` 或 `SUPPORT` 发起查询
2. 系统校验 `task_type`、`task_status` 与 `limit`
3. 按时间倒序返回 `task_log`

### 4.3 异常看板

系统从以下来源聚合最近异常：

- `notification.push_status = FAILED`
- 每个 `(task_type, biz_key)` 的最新失败任务
- `payment_callback_log.callback_status = REJECTED`
- `operation_log.module_name = 'ops'` 且 `operation_name LIKE 'mark_exception:%'` 的手工标记

### 4.4 通知失败重试

1. `ADMIN` 指定单条通知 ID 或批量 limit
2. 系统读取 `notification`
3. 当前仅对 `OUTBID` 且 `biz_type = AUCTION` 的失败通知执行直接推送重试
4. 成功则把 `push_status` 改为 `SENT`，失败则保持 `FAILED`
5. 每次重试都会向 `task_log` 追加一条 `NOTIFICATION_PUSH` 记录，并累加 `retry_count`

### 4.5 手工异常标记

1. `ADMIN` 或 `SUPPORT` 提交异常类型、业务键和值班备注
2. 系统向 `operation_log` 写入 `module_name = 'ops'`
3. `operation_name` 采用 `mark_exception:<EXCEPTION_TYPE>`

说明：

- Redis 降级使用 `REDIS_DEGRADED`
- 业务键建议写具体对象，如 `bid_cache_auction_123`

### 4.6 补偿入口

当前支持以下补偿类型：

| 补偿类型 | 实际动作 |
|---|---|
| `NOTIFICATION_PUSH` | 失败通知重试 |
| `ORDER_SETTLEMENT` | 调用 `OrderService::GenerateSettlementOrders()` |
| `ORDER_TIMEOUT_CLOSE` | 调用 `OrderService::CloseExpiredOrders()` |
| `STATISTICS_DAILY_AGGREGATION` | 调用 `StatisticsService::RebuildDailyStatistics(stat_date)` |

## 5. 服务接口

当前已实现的核心服务位于 `src/modules/ops/ops_service.h`。

### 5.1 `ListOperationLogs`

入参：

- `authorization_header`
- `OperationLogQuery`
  - `module_name`
  - `result`
  - `limit`

返回：

- `OperationLogQueryResult.records`

### 5.2 `ListTaskLogs`

入参：

- `authorization_header`
- `TaskLogQuery`
  - `task_type`
  - `task_status`
  - `limit`

返回：

- `TaskLogQueryResult.records`

### 5.3 `ListSystemExceptions`

入参：

- `authorization_header`
- `SystemExceptionQuery.limit`

返回：

- `SystemExceptionQueryResult.records`
  - `source_type`
  - `source_id`
  - `biz_key`
  - `current_status`
  - `summary`
  - `detail`
  - `occurred_at`
  - `retryable`

### 5.4 `MarkException`

入参：

- `authorization_header`
- `MarkExceptionRequest`
  - `exception_type`
  - `biz_key`
  - `detail`

返回：

- `MarkExceptionResult`

### 5.5 `RetryFailedNotifications`

入参：

- `authorization_header`
- `NotificationRetryRequest`
  - `notification_id`
  - `limit`

返回：

- `NotificationRetryResult`
  - `scanned`
  - `succeeded`
  - `failed`
  - `skipped`
  - `affected_notification_ids`

### 5.6 `RunCompensation`

入参：

- `authorization_header`
- `CompensationRequest`
  - `compensation_type`
  - `biz_key`
  - `limit`

返回：

- `CompensationResult`
  - `compensation_type`
  - `scanned`
  - `succeeded`
  - `failed`
  - `skipped`
  - `affected_keys`

## 6. 错误码

| 错误码 | 含义 | 场景 |
|---|---|---|
| `4108` | `auth permission denied` | `USER` 访问运维模块 |
| `4901` | `ops query limit invalid` | `limit` 超出 `1-100` |
| `4902` | `ops compensation type invalid` | 补偿类型不支持 |
| `4903` | `ops exception type invalid` | 异常类型格式不合法 |
| `4904` | `ops biz key invalid` | 业务键缺失或格式不合法 |
| `4905` | `ops mark detail invalid` | 异常备注为空或过长 |
| `4906` | `ops query token invalid` | 日志过滤条件格式不合法 |

## 7. 关键设计决定

- 决定 1：不新建异常表，统一复用 `operation_log`、`task_log`、`notification`、`payment_callback_log`
  原因：课程设计阶段优先保证可追踪和可复用，不重复造外围事实

- 决定 2：异常看板只把“当前失败通知”“每个任务键的最新失败记录”“被拒绝支付回调”“手工异常标记”作为最小观测集
  原因：这样能避免把历史已恢复的通知失败长期误判为未处理异常

- 决定 3：通知重试只重试 `OUTBID` 直推失败
  原因：站内通知本身已经持久化，当前只有竞价超越提醒存在失败后可安全重推的实时直推语义

- 决定 4：补偿入口只调用既有服务，不在运维层重写结算逻辑
  原因：补偿层应复用主模块的状态机和事务边界，避免出现第二套规则
